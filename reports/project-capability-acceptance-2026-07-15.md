# News Codex 项目能力总验收（2026-07-15）

## 1. 总体结论

News Codex 已经形成一条可运行、可追溯的本地情报链路：

```text
Provider/Target 目录
  → 来源探测
  → 受控抓取与 Worker
  → RawItem 规范化
  → 事件筛选、聚类与质量评分
  → 中文网页查看与证据下钻
```

当前产品定位应收口为：**可供个人使用的 AI/科技信息发现与事件研判工作台**。

它已经不再只是“抓几个官网 RSS”的原型。真实数据库中已经包含社交、社区、专业媒体、聚合发现、官方一手和研究来源，并能在网页中查看来源、抓取批次、原始条目和事件证据。

但它还不是无人值守的“自动新闻雷达”：当前 `main` 运行时未加载 MiniMax Key、Worker 没有近期任务心跳、实体数据为空，也尚未提供稳定的定时抓取、中文日报、重大提醒和事实确认工作流。

### 为什么“30 个可试用”却有“82 个抓取过”

这两个数字不是同一时间口径：

- `30` 是按当前严格资格规则重新计算的可试用 Target。
- `82` 是历史上至少成功完成过一次 FetchRun 的当前 Target。
- 来源定义、资格规则和接入方式在开发过程中发生过变化，因此历史成功不能冒充当前资格。

网页已经对这两个指标分别命名，但后续还应补充“当前最近一次抓取状态”和“历史曾成功”两个独立筛选。

### 3.2 Target 覆盖方式

当前 187 个 YAML Target：

| 覆盖方式 | 数量 | 含义 |
| --- | ---: | --- |
| 直接获取 | 69 | 项目可直接访问其 RSS、Atom 或公开/正式 API |
| 间接发现 | 58 | 通过聚合、搜索或替代公开入口发现，再回到原始来源确认 |
| 仅登记 | 60 | 已知道该目标重要，但当前没有合规、低成本的自动获取路径 |

数据库中的来源获取方式覆盖 RSS、HTML、REST API、公共 API 和 Atom。HTML 目标仍受人工批准和访问策略约束；“登记了 HTML URL”不代表允许自动抓取。

26 个失败中 RSS 占 23 个，是下一轮最值得集中排查的技术问题；另外包含少量 Atom 和公共 API 失败。

当前产出较多的平台包括 Bluesky、Mastodon、Google News、Hacker News、Techmeme、arXiv、YouTube 和 GitHub，也已经覆盖 The Verge、WIRED、TechCrunch、Ars Technica、BBC、CNBC、Guardian、MIT Technology Review 等媒体入口。

因此当前系统已经可以用于：

- 发现 AI 产品、模型、开发工具和研究论文变化；
- 观察 Bluesky、Mastodon、Hacker News 等社区或社交信号；
- 通过 Google News、Techmeme、GDELT 扩大媒体发现面；
- 通过 GitHub Releases、YouTube 频道、arXiv 获取开发、视频与研究内容；
- 在事件详情中查看来源性质、独立证据根、质量评分和固定运行快照。

当前还不能保证：

- X、Facebook、Instagram、TikTok、LinkedIn 的免费实时全量覆盖；
- 每个聚合结果都已经由两个独立专业来源确认；
- 每个事件都完成了中文模型增强；
- RawItem 列表中的跨 Target 重复内容已经完全合并；
- 系统在电脑重启后自动恢复定时抓取和 Worker。

## 5. 事件与 MiniMax 状态

### 5.1 事件质量链路

最近 72 小时共有 480 条 RawItem 进入质量流程，全部已有确定处理结果：

- 纳入 266 条；
- 排除 214 条；
- 形成 225 个候选簇；
- 主要排除原因是无 AI 信号（146 条），其次是泛科技、歧义词、广告订阅和缺少事件动作。

默认 `/events` 页面展示最近一次有效 Operation 的固定快照，而不是随数据库 current 指针变化的混合结果。这解决了事件列表与详情证据不一致的问题。

### 5.2 MiniMax

历史数据库有 275 条 MiniMax 使用记录：

- 2 次成功；
- 72 次结构修复重试；
- 201 次规则回退；
- 全部使用 `MiniMax-M2.7-highspeed`；
- 273 次用于事件中文增强，2 次用于事件对比较验。

当前根目录 `.env` 没有有效 `MINIMAX_API_KEY`，所以 `main` 网页正确显示“MiniMax 当前未配置”。旧的 RawItem 工作树中存在本地配置，但本次验收没有读取、打印、复制或修改密钥。

结论：MiniMax 适配器和审计链路已经实现，但当前主运行时没有完成配置收口，不能把 275 条调用记录理解为 275 次有效增强。

## 7. 已确认问题

### P0：运行闭环必须收口

1. **MiniMax 主运行时未配置**  
   根目录环境没有加载 Key，最近事件处理大量回退。需要在主运行时安全配置并执行一次官方 API 健康检查和小样本事件增强验收。

2. **Worker 当前无近期任务活动**  
   `/system` 显示 `stale`，队列为 0。需要明确“手动按需启动”还是“本机常驻”，并提供一个统一启动/停止/状态入口，避免用户不知道任务是否会被消费。

3. **目录存在 2 个数据库残留**  
   `legacy-source` 和 `universe-youtube-1` 不在当前 YAML。应通过一次显式、可审计的目录对账处理，不能静默删除数据库记录。

### P1：来源可用性收口

1. 对 21 个未探测 Target 完成第一轮能力或内容探测。
2. 集中排查 23 个 RSS 最新失败目标，按 URL 失效、网络、格式和重定向分类。
3. 将 65 个 HTML 策略阻塞与技术失败在报告和页面中进一步分开，避免用户误解为“来源坏了”。
4. 诊断 3 个 Reddit Target 的 OAuth `permission_required`，无权限时保持明确阻塞，禁止回退到登录 Cookie 抓取。
5. 为 82 个历史成功来源增加“最近一次成功时间”和“连续成功轮次”口径，避免历史成功掩盖当前退化。

### P2：情报产品能力

1. 实体抽取和公司/人物/模型/技术下钻；
2. 跨 Target 重复信息的展示合并；
3. 事件事实确认状态与独立证据门槛；
4. 中文日报、重要事件提醒和用户关注主题；
5. 最后再实现推荐与个性化，不应提前改写来源和事件底层。

## 8. 下一里程碑

下一里程碑应命名为 **v1.2 运行闭环收口**，不再扩写架构，也不立即新增更多来源。

建议一次性完成：

1. 把 MiniMax 配置安全迁移到 `main` 本地运行环境，只验证布尔状态和调用结果，不记录 Key；
2. 完成一次 MiniMax 官方 API 小样本调用，确认成功、token、延迟和回退路径；
3. 提供明确的本机 Web + Worker 启动、停止和状态检查流程；
4. 处理两个目录漂移 ID；
5. 对 21 个未探测和 23 个 RSS 失败目标执行一轮集中探测；
6. 生成新的当前来源健康快照；
7. 在 `/sources` 中把历史成功、最近成功和当前阻塞口径区分清楚。

完成 v1.2 后，News Codex 才适合进入下一波“更多来源目标扩充”，然后再做中文日报与提醒。
