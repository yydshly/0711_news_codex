# 中文日报完整性修复与现有页面收口设计

## 背景与问题

日报 5 已经具备“今日决策简报”“情报全览”“完整条目与证据”三层内容，也分别支持决策版和全览版语音。本次不重新设计日报系统。

当前问题来自一次 Windows PowerShell 向 Python 标准输入传递中文常量的批量审核操作。部分中文在进入应用前已经被替换为 ASCII 问号，随后作为合法非空文本写入数据库。日报 5 的 49 条全览候选中有 29 条最新人工审核记录包含连续问号，其中 17 条进入全览正文和全览语音。原始事件快照大多正常，决策版语音脚本也正常；损坏集中在后写入的全览审核标题、文章概述、审核建议和证据评价。

## 目标

- 使用现有修订机制产生日报 5 的新修订版，不修改已归档历史记录。
- 修正损坏的中文审核字段，并重点复核进入全览的条目。
- 重新形成可读的决策简报、情报全览、完整报告内容以及两种独立音频。
- 在保存审核、归档和语音合成三个边界阻止同类损坏再次进入成品。
- 只整理现有详情页的信息层级、换行和定位，不增加新页面或重写查询架构。

## 不在本次范围内

- 不重新抓取 41 个高价值目标，不重新运行来源探测或事件管线。
- 不改变 187 个来源目标的状态和来源合法性结论。
- 不让 MiniMax 决定条目是否可信、是否合法或是否收录。
- 不修改已归档日报 5 及其既有音频制品；历史制品只作为审计记录保留。
- 不开发定时日报、推送、推荐或复杂摘要。
- 不拆分新的日报路由，不建立第二套报告模型。

## 方案比较

### 方案 A：直接修改日报 5 的数据库记录

页面能最快恢复，但会破坏归档不可变性、审核历史和既有音频哈希，不能采用。

### 方案 B：只创建修订版并人工修复数据

可以恢复当前日报，但无法阻止下一次编码错误再次进入数据库或语音，闭环不足。

### 方案 C：修订版恢复 + 三层完整性防线 + 现有页面小幅整理（采用）

保留追加式审核和归档不可变性；在输入、归档、合成三个阶段识别异常问号；页面继续使用现有单页结构，并清晰标出历史损坏和新版内容。该方案改动集中、可测试，也不重复已有能力。

## 文本完整性规则

新增一个无数据库依赖的日报文本完整性函数。第一阶段只识别明确的编码损坏特征：同一字段中出现连续 4 个或更多 ASCII `?`。正常的英文问句、单个 ASCII 问号和中文全角问号 `？` 不受影响。

稳定错误码使用 `daily_report_text_corrupted`，中文诊断为“检测到疑似编码损坏的连续问号，请修正中文内容后再继续。”

防线分为三层：

1. `DailyReportEditorialReviewDraft` 和 `DailyReportOverviewEditorialReviewDraft` 保存前校验四个中文字段，拒绝新增损坏记录。
2. 日报归档前扫描决策条目和全览条目的最新审核记录；任一可见审核字段损坏即拒绝归档。排除和重复条目仍属于完整报告审计内容，因此也必须通过校验。
3. Worker 在创建音频制品和调用 MiniMax 之前校验最终脚本。这样旧归档日报即使含历史损坏，也不会再次生成错误音频或消耗语音额度。

输入校验、归档校验和 Worker 错误都映射为稳定中文页面诊断。结构化日志只记录日报 ID、渲染版本和错误码，不输出完整审核文本或密钥。

## 日报 5 的安全修复

修复遵循现有不可变设计：

1. 从日报 5 创建下一修订版，复制固定事件快照、条目顺序、审核结论和重复关联。
2. 对新修订版中检测为损坏的 29 条记录追加新的审核版本，不覆盖复制记录。
3. 中文标题以固定快照标题为基础；中文文章概述以固定快照概述为事实边界，不补写快照中不存在的事实。
4. 审核建议根据 `keep`、`needs_evidence`、`exclude`、`duplicate` 的既有结论重写为具体中文动作；证据评价只使用固定快照中的公开证据、独立证据根数、确认说明和限制。
5. 重点逐项复核进入全览的 17 条，发现标题夸大、日期不符或证据不足时允许把结论降为 `needs_evidence` 或 `exclude`；MiniMax 不参与该判断。
6. 修复后再次扫描全部决策审核、全览审核、决策脚本和全览脚本，连续问号计数必须为 0。
7. 新版归档后分别生成决策版和全览版音频；两种音频使用各自脚本和制品，不互相覆盖。

一次性数据修复通过 UTF-8 文件执行，不再把含中文常量的脚本经 PowerShell 管道传给 Python。修复脚本只服务本次数据操作，不作为新的抓取器或日报子系统。

## 页面结构

保留 `/daily-reports/{report_id}` 和现有模板，只做以下整理：

1. 页面顶部增加三个页内定位入口：“决策简报”“情报全览”“完整报告与证据”。
2. “今日决策简报”继续展示已进入决策版的中文标题、文章概述、审核建议、证据评价、决策播报稿和决策版音频。
3. “情报全览”继续展示 `keep` 与 `needs_evidence` 条目，按已确认、热点、新兴信号分组，并展示全览播报稿和全览版音频。
4. 现有“今日确认要闻”“值得关注的线索”合并到一个有明确总标题的“完整报告与证据”区域；内容、公开证据、审核历史和草稿编辑能力不删除。
5. 全部候选审核清单继续保留在情报全览区域，排除和重复项不进入正文或音频，但必须显示中文原因、下一步动作和重复关联。
6. 长标题、中文概述、审核建议、证据评价和 URL 必须自然换行；桌面与窄屏均不产生横向溢出。
7. 检测到历史损坏时，页面顶部显示中文完整性警告，说明该归档版不可直接作为最终报告，并引导创建修订版。

页面不会改成三个独立路由，也不会用新的前端框架。页内定位只增强阅读顺序，不改变现有表单、动作令牌、归档、修订或音频接口。

## 数据流与错误处理

正常数据流保持为：固定事件快照 → 日报草稿 → 人工审核 → 归档 → 决策/全览脚本 → 持久语音任务 → MiniMax 音频制品。

文本完整性校验只负责阻止明确损坏，不负责判断新闻真实性。真实性仍由人工审核结论和证据状态表达。单条审核失败只影响当前表单；归档失败保持草稿；音频校验失败产生不可重试的中文任务结果，不调用外部接口。现有超时、有限重试、取消、心跳、恢复和结构化日志机制保持不变。

## 测试驱动与验收

实现严格使用测试驱动：先增加失败测试并确认失败原因，再写最小实现。

自动化测试覆盖：

- 连续 4 个 ASCII 问号被识别，1 至 3 个 ASCII 问号和中文问号不误报；
- 两类审核草稿的四个文本字段均拒绝损坏内容；
- 归档扫描最新审核版本，同时覆盖决策、全览、排除和重复条目；
- 检测失败时日报仍为草稿，且不会创建语音操作；
- Worker 在调用 MiniMax 前拒绝损坏脚本，不创建成功音频文件；
- 详情页显示三段定位、两种独立报告与音频、完整报告证据区以及历史损坏警告；
- 现有归档、修订、审核历史、全览筛选和音频测试不回归。

真实数据验收覆盖：

- 日报 5 保持字节级不可变并显示历史损坏提示；
- 新修订版中损坏审核字段数为 0；
- 决策脚本和全览脚本中的连续问号数均为 0；
- 页面可直接看到决策简报、情报全览和完整报告与证据；
- 决策版和全览版音频分别生成并可播放；
- 最终执行完整 `pytest`、`ruff check .` 和本地真实网页验收。

未经用户确认不合并、不推送；不读取或输出 `.env`，不触碰用户保留的报告文件。
