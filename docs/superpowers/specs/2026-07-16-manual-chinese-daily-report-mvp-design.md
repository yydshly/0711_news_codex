# 手动中文日报 MVP 设计

## 1. 目标

在不重新抓取来源、不重新聚类事件、也不依赖 MiniMax 的前提下，使用最近一次完整事件运行快照生成一份可追溯、可人工整理、可归档的中文日报。

日报必须严格区分：

- **今日确认要闻**：只包含 `confirmed` 事件，可以作为已确认信息阅读；
- **值得关注的线索**：只包含 `emerging` 事件，并逐条醒目标注“尚未确认”。

第一版提供手动生成、草稿整理和历史存档，不提供定时生成、推送、推荐或自动发布。

## 2. 已有能力与复用边界

项目已经具备：

- 持久化 `RawItem`、事件、事件版本、事件评分和证据关系；
- 完整事件运行快照，以及按 `operation_id + event_id + version_number` 固定读取事件的能力；
- 中文标题、中文摘要、关注原因、确认状态、独立证据根数量和来源时间线；
- MiniMax 不可用时的规则中文回退；
- CSRF 防护、受控网页操作和 PostgreSQL 迁移体系。

日报直接复用这些能力，不创建第二套抓取、聚类、确认或中文生成管线。日报生成只读取已经完成且可见的事件运行快照。

## 3. 不在本阶段实现

- 不重新探测或抓取来源；
- 不重新运行事件聚类、合并、确认或模型增强；
- 不增加新 Provider 或 Target；
- 不调用 MiniMax 或其他外部模型；
- 不实现定时日报、邮件、微信、Slack 或其他推送；
- 不实现个性化推荐、自动发布或多人协作编辑；
- 不允许日报操作修改 `Event`、`EventVersion`、`RawItem` 或来源证据；
- 不把 `emerging` 事件改写或展示为已确认事实。

## 4. 方案比较与选择

### 4.1 选择：数据库存档版

每次生成一条日报记录，并为入选事件保存固定的日报条目快照。草稿允许人工排除和排序，归档后不可修改。若归档后需要调整，应基于原日报创建一个新修订版。

优点：历史稳定、网页管理方便、证据可追溯，后续增加定时任务或推送时不需要更换数据模型。

### 4.2 未选择：实时网页版

实时读取当前事件开发量较小，但昨日页面会随着事件 current 指针变化，无法满足历史日报不可变要求。

### 4.3 未选择：仅 Markdown 文件

文件便于阅读，但不适合安全的草稿编辑、条目排序、状态控制和网页存档，也难以复用数据库中的事件版本关系。

## 5. 数据模型

### 5.1 `daily_reports`

保存一份日报的身份、窗口、来源快照和生命周期：

- `id`：主键；
- `report_date`：按 `Asia/Shanghai` 计算的日报日期；
- `timezone`：固定为 `Asia/Shanghai`；
- `window_hours`：仅允许 `24`、`48` 或 `72`；
- `window_start`、`window_end`：UTC 时间；
- `source_operation_id`：生成时使用的完整事件运行 `OperationRun`；
- `status`：`draft` 或 `archived`；
- `revision`：同一 `report_date + window_hours` 下从 1 开始递增；
- `supersedes_report_id`：可选，指向被修订的已归档日报；
- `generated_at`、`archived_at`：审计时间；
- `generation_summary`：只保存无敏感信息的计数和降级说明。

唯一约束为 `report_date + window_hours + revision`。数据库约束和服务层同时阻止修改已归档日报。

### 5.2 `daily_report_items`

保存日报中每个事件的固定展示快照：

- `id`、`daily_report_id`；
- `event_id`、`event_version_number`；
- `section`：`confirmed` 或 `emerging`；
- `position`：分区内顺序；
- `included`：草稿中是否保留；
- `snapshot`：生成时固定的无敏感信息 JSON。

`snapshot` 至少包含：

- 中文标题、中文摘要、为什么值得关注；
- 事件状态、展示层级、分类和排名分数；
- 独立证据根数量和确认说明；
- 数据截止时间和事件发生时间；
- 经过公开 URL 清洗的证据列表：标题、来源角色、原始链接、发布时间、是否独立、已知限制；
- 中文生成来源：模型结果或规则回退；
- 明确的 `unconfirmed=true` 标记（仅 `emerging`）。

条目唯一约束为 `daily_report_id + event_id + event_version_number`。历史页面只读取条目 `snapshot`，不跟随 Event current 指针或可变 RawItem 字段。

## 6. 生成规则

### 6.1 输入快照

生成服务读取最近一次完整、读者可见的事件运行快照。不得退回到全局 current 事件目录冒充某次运行结果。

如果没有完整事件运行快照，生成失败并显示中文诊断：“尚无完整事件运行快照，请先完成事件构建。”该失败不得创建空日报。

### 6.2 时间窗口

用户可选择最近 `24`、`48` 或 `72` 小时，默认 `24` 小时。`window_end` 使用被选事件运行快照的固定截止时间，而不是网页请求的当前时间；`window_start = window_end - window_hours`。

事件按快照内的事件发生时间过滤。缺失发生时间的事件不自动进入日报，并计入 `generation_summary.skipped_missing_time`。

### 6.3 今日确认要闻

只纳入：

- `status == confirmed`；
- 发生时间处于日报窗口；
- 快照中存在中文标题和摘要；
- 事件版本和评分记录完整。

按 `rank_score`、`occurred_at`、`event_id` 稳定排序，最多 20 条。少于 20 条时不补入 emerging。

### 6.4 值得关注的线索

只纳入：

- `status == emerging`；
- `display_tier` 为 `hotspot` 或 `signal`，排除 `audit_only`；
- 发生时间处于日报窗口；
- 快照中存在中文标题和摘要；
- 事件版本和评分记录完整。

按 `rank_score`、`occurred_at`、`event_id` 稳定排序，最多 20 条。每个线索在数据和页面中都必须带有“尚未确认”标记，不得仅依赖颜色区分。

### 6.5 空分区

确认要闻或线索允许为空。页面明确显示“本时间窗口暂无已确认要闻”或“本时间窗口暂无入选线索”，不得为了填充版面降低确认门槛。

## 7. 草稿与归档

### 7.1 手动生成

`POST /daily-reports` 接收 `window_hours` 和可选的 `supersedes_report_id`。这是本地数据库短事务：不访问网络、不调用模型、不启动来源任务。

相同用户操作令牌和相同生成参数必须幂等，避免重复点击产生多个相同修订版。生成成功后跳转到新日报详情页。

### 7.2 草稿操作

草稿允许：

- 排除或恢复某个条目；
- 在同一分区内上移或下移条目；
- 归档日报。

不允许把条目从 `emerging` 移到 `confirmed`，不允许编辑事件标题或摘要，不允许添加不属于固定运行快照的事件。这样可避免日报成为绕过证据规则的第二套编辑系统。

### 7.3 归档

归档操作固定最终顺序和 `included` 状态，写入 `archived_at`。归档后所有变更端点返回冲突诊断，不做隐式复制或修改。

如需调整已归档日报，用户显式选择“创建修订版”。系统复制已归档日报的固定条目快照到新草稿，将 `revision + 1`，并写入 `supersedes_report_id`。旧日报保持不变。

## 8. 网页

### 8.1 导航

在现有侧边导航增加“中文日报”，不改变来源、事件和系统页面的职责。

### 8.2 `/daily-reports`

展示：

- 生成日报表单：24/48/72小时；
- 日报存档列表：日期、修订版、状态、数据截止时间、确认要闻数、未确认线索数；
- 当前是否存在完整事件快照；
- 明确提示“生成日报不会重新抓取或调用模型”。

### 8.3 `/daily-reports/{id}`

页面顶部显示：

- 日报日期、状态和修订版；
- 数据截止时间、事件运行编号和时间窗口；
- MiniMax 是否降级；
- “确认要闻”和“未确认线索”计数。

正文严格分为“今日确认要闻”和“值得关注的线索”。线索区整体及每条内容都显示“尚未确认”。每条显示固定中文内容、关注原因、证据数量、来源链接、发布时间和限制。

草稿显示排除、恢复、排序和归档操作；归档页面只读，并提供“创建修订版”。

## 9. 安全与错误处理

- 所有写操作使用现有 CSRF/动作令牌；
- 所有 ID 和状态转换在服务层校验；
- 只允许 `http`、`https` 且无用户名、密码、查询密钥的公开证据 URL 进入日报快照；
- 不保存提示词、模型原始响应、API 错误正文、Cookie、凭据或环境变量值；
- 数据库写入使用短事务，唯一约束处理并发重复生成；
- 单个事件版本缺失或损坏时跳过该事件并记录中文计数，不阻止其余条目生成；
- 若所有候选事件均损坏，仍可生成明确为空的日报，但必须在运行说明中记录跳过数量；
- 数据库不可用时复用现有中文错误页面，不泄露连接串。

## 10. 组件边界

- `newsradar.daily_reports.schema`：日报状态、分区和纯输入/输出类型；
- `newsradar.daily_reports.repository`：日报与条目持久化、并发约束和归档状态转换；
- `newsradar.daily_reports.service`：选择完整事件快照、过滤排序、生成固定条目快照和修订；
- `newsradar.web.daily_report_queries`：列表及详情只读投影；
- `newsradar.web.app`：薄路由、动作令牌、参数校验和重定向；
- `daily_reports.html`、`daily_report_detail.html`：现有网页样式下的列表与详情。

日报服务依赖已有事件运行快照读取能力，不直接依赖抓取器、MiniMax 客户端或 Worker 网络执行器。

## 11. 数据流

1. 用户在日报列表选择 24/48/72 小时并提交；
2. 服务定位最近一次完整事件运行快照；
3. 服务按固定截止时间和状态/层级规则筛选事件版本；
4. 服务把显示所需内容和公开证据复制为日报条目快照；
5. 数据库在一个短事务中创建日报和条目；
6. 用户在草稿中排除、恢复或排序；
7. 用户归档，日报变为不可变历史记录；
8. 页面始终读取日报条目快照，不跟随事件后续变化。

## 12. 测试策略

### 12.1 纯规则测试

- confirmed 和 emerging 永不跨区；
- `audit_only` emerging 不入选；
- 24/48/72 小时窗口边界固定；
- 稳定排序和20条上限；
- emerging 始终生成 `unconfirmed=true`；
- 缺失时间、版本或评分的事件按规则跳过。

### 12.2 Repository 与迁移测试

- 新表、索引、外键和唯一约束可升级与降级；
- 并发重复生成不会创建相同修订版；
- 草稿可排除、恢复和排序；
- 归档后拒绝修改；
- 修订版复制固定快照且不改变旧日报。

### 12.3 网页测试

- 列表、详情、空状态和404；
- 动作令牌保护所有 POST；
- 线索区和每条线索都有文字“尚未确认”；
- 归档后不显示编辑按钮；
- 页面不包含敏感 URL、模型原文或凭据；
- 生成操作不调用抓取器、Worker 网络处理或 MiniMax。

### 12.4 真实验收

- 使用现有 PostgreSQL 数据生成一份24小时中文日报；
- 验证来源事件运行编号和版本均可追溯；
- 记录确认要闻数、未确认线索数和跳过数；
- 归档日报后再运行或修改事件，确认历史日报内容不变；
- MiniMax 未配置或不可用时仍能生成；
- 运行完整 `pytest`、Ruff、迁移测试和真实浏览器验收。

## 13. 验收标准

- 用户可以手动生成、整理、归档和回看中文日报；
- 生成日报不触发来源探测、抓取、事件聚类或模型调用；
- confirmed 和 emerging 严格分区，所有 emerging 明文标注“尚未确认”；
- 日报固定到完整事件运行和事件版本，归档内容不随 current 事件变化；
- 每条内容都能追溯到公开证据链接和事件版本；
- 单条损坏数据不阻塞整份日报；
- 归档不可变，修订生成新记录；
- MiniMax 降级不影响基础日报生成；
- 完整测试、静态检查、迁移和真实网页验收全部通过。
