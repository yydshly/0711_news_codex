# 系统网络继承设计

## 目标

让 News Codex 的来源研究探测与正式抓取自动遵循运行该进程的 Windows 网络环境：系统代理、VPN 虚拟网卡、全局模式或规则模式。用户不需要在 News Codex 中填写代理地址。

## 背景与边界

当前研究探测客户端以 `trust_env=False` 创建，显式忽略系统代理和代理环境变量。因此浏览器可以经 VPN 访问的来源，News Codex 仍可能直连失败。

本设计只改变 HTTP 客户端是否继承系统网络配置；不实现代理管理能力。具体包括：

- 不增加代理 URL、端口、用户名或密码配置项。
- 不把代理地址、`Proxy-Authorization`、Cookie、授权头或环境变量写入数据库、日志、网页或 YAML。
- 不使用浏览器登录态、Cookie、代理轮换、验证码破解或规避平台限制。
- 仍只使用已经审核的 HTTPS 目标、已有的响应大小限制、超时、限流与重试策略。
- 若 VPN 为虚拟网卡或全局隧道，程序自然沿用操作系统路由；若为规则模式，则由每个真实来源的探测结果决定可达性。

## 方案选择

采用 HTTPX 的系统环境继承机制，而非在 News Codex 内保存或配置代理。

- 研究探测客户端允许读取系统网络设置；仍保持无 Cookie、显式跳转校验和无凭据请求。
- 正式抓取客户端使用相同的系统网络继承语义。
- MiniMax 客户端不在本次范围内改变；它已有独立的外部服务配置和调用边界。

这样同时支持 Windows 系统代理、代理环境变量及 VPN 虚拟网卡路由，又不需要猜测用户使用的 VPN 模式。

## 组件与数据流

1. `Settings` 增加一个默认开启的布尔开关，名称为 `http_trust_env`，对应环境变量 `HTTP_TRUST_ENV`。它只允许 `true` 或 `false`，用于在少数排障场景显式关闭系统网络继承。
2. `new_safe_probe_client()` 从该设置读取 `trust_env`。客户端继续不携带 Cookie，继续禁止自动跳转，并继续通过 `safe_get()` 校验每一跳。
3. `HttpPolicy.default()` 从同一设置读取 `trust_env`，使 Worker 的正式抓取与研究探测网络路径一致。
4. 网络/探测结果只记录既有的分类错误（如 `connection_error`、`timeout`、`http_429`），不记录代理地址或网络环境细节。
5. 网页增加只读的中文网络提示：说明探测与抓取会跟随本机系统网络；展示“系统网络继承已启用/已关闭”，但不显示系统是否存在代理、代理协议、地址或端口。

## 错误处理与可观察性

- 连接失败、超时、TLS 失败、429 和 HTTP 5xx 继续按照既有探测/抓取错误路径写入运行记录。
- 页面将现有运行失败原因转为中文诊断；网络提示不能把任意失败断言为“代理问题”，因为规则模式、VPN 路由、DNS、目标站故障和限流都可能产生类似结果。
- `HTTP_TRUST_ENV=false` 是排障开关，默认不写入 `.env.example` 以外的任何用户配置。默认值为 `true`。

## 验证与验收

- 单元测试确认研究探测和正式抓取客户端默认使用 `trust_env=True`，并在 `HTTP_TRUST_ENV=false` 时关闭。
- 单元测试确认研究探测仍拒绝不安全 URL、认证候选及携带 Cookie 的客户端。
- 单元测试确认网页只渲染“已启用/已关闭”状态，不渲染 `HTTP_PROXY`、`HTTPS_PROXY` 或代理 URL。
- 在当前电脑网络环境下，对 OpenAI RSS、Hacker News、arXiv 与至少一个社交/社区公开来源运行真实探测；报告每个来源的成功或失败分类，不把失败来源误标为已覆盖。
- Worker 领取至少一个开放来源抓取任务并完成或以可读错误结束，证明其使用与研究探测相同的网络继承配置。

## 非目标

- 不新增代理配置页、账号体系、抓取登录页、动态网页爬取、摘要、推荐或调度器。
- 不保证所有来源在任意 VPN 规则下可达；规则模式下的可达性必须以当次探测为准。
