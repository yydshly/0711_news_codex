# 来源失败修复批次设计

## 目标

对来源全量初探中最新内容探测未成功的 Target 建立可审计、可恢复且合规的修复流程。目标不是重复扫描全部目录，也不是放宽安全限制；目标是为每个失败 Target 给出可验证的失败归因、首选或备用获取方式，以及明确的升级或停止结论。

本批次以 2026-07-13 的 `source-trial-baseline.md` 为基线。该基线有 166 个已探索 Target，其中 27 个最新探测为未成功。启动修复批次时，从 PostgreSQL 中按每个来源最新、已完成的内容探测记录生成不可变批次清单（Target ID、原探测 ID、基线时间和原结果），而不是把 ID 写死在代码中。后续新探测不会让待处理 Target 从本批报告中消失。

## 非目标与硬边界

- 不重新探测全部 166 个来源。
- 不使用登录态、浏览器 Cookie、验证码绕过、代理轮换、IP 绕过、非官方私有接口、无头浏览器、JavaScript 执行或模拟用户交互。
- 不将能力探测、目录登记、间接发现或 HTML 页面可访问误表述为可持续内容抓取。
- 不实现摘要、推荐、推送或定时调度。
- 不读取、记录或展示 API Key、数据库连接串、Cookie、代理地址、授权头或完整的敏感响应体。
- MiniMax 可以在失败说明中提供可选的中文措辞，但不能改变失败分类、访问方式批准、重试策略或来源启用结论；模型不可用时完整流程必须照常运行。

系统继续继承用户本机的网络/VPN 路径（`HTTP_TRUST_ENV`），但不得分析、保存或修改该路径。某域名在当前网络不可达，只能记录为本次不可达。本批次复用现有安全探测客户端的 HTTPS、无凭据、无 Cookie、显式跳转和响应大小限制；不引入代理探测、代理配置或 DNS 固定 IP 机制。

## 处置模型

每个失败 Target 依序进入以下状态机：

```text
最新内容探测未成功
  -> 归因分类
  -> 官方方式复核
  -> 有界候选验证（仅已允许的方式）
  -> 可试用（RSS/API）/ HTML 研究候选 / 保留仅发现 / 受限目录 / 待复核
```

### 1. 失败归因

归因器只消费已持久化的非敏感探测证据：结果、HTTP 状态、错误代码、延迟、样本数量、字段完整度、响应类型和结构指纹。每个 Target 必须得到一个主分类和一个中文建议：

| 分类 | 判断 | 动作 |
| --- | --- | --- |
| `network_transient` | 超时、TLS/DNS、5xx 或当前网络不可达 | 保留证据，低频单次复测；不得自动换网络或代理 |
| `rate_limited` | 429 或明确的 Retry-After | 记录最早复查时间并停止；本批次不等待、不循环重试 |
| `endpoint_changed` | 404、无效 feed、错误内容类型、解析结构漂移 | 查官方主页、RSS/Atom、Sitemap 或官方 API 文档 |
| `content_incomplete` | 成功响应但无样本、缺标题/规范 URL/日期 | 修复字段映射或寻找同一官方来源的备用端点 |
| `authentication_or_policy` | 401、403、登录、Cookie、验证码、付费或审批要求 | 停止自动修复，保留受限目录和解锁条件 |
| `unknown` | 证据不足或错误格式无法识别 | 待人工复核，不猜测原因 |

### 2. 官方方式复核与优先级

每个候选方式均须关联官方身份 URL、文档/条款/页面证据 URL 和审核时间。优先级固定如下：

1. 官方 RSS 或 Atom；
2. 官方公开 API；
3. 经过审计的、官方公开 HTML 研究候选；
4. 仅发现来源或受限目录。

不能为获得更多覆盖而把间接聚合源改成事实来源，也不能将需要凭据的平台静默回退到网页抓取。

### 3. HTML 白名单候选

HTML 可作为研究候选，但不是自动抓取的默认回退，本批次也不新增生产 HTML Fetcher。它必须同时满足：

- 目标属于已确认的官方机构或专业媒体，HTTPS 且身份 URL 同域或有明确关联；
- robots 指示和可核实条款不禁止该自动读取用途；
- 不需要登录、Cookie、验证码、浏览器会话或付费墙规避；
- 入口、允许重定向域和解析选择器预登记；每一跳只能到同源或显式审核过的 HTTPS 域名；
- 仅接受静态 HTML，不执行 JavaScript、不启动浏览器、不提交表单；
- 提取范围限于标题、规范 URL、发布时间、作者和摘要等必要字段，不镜像全文；
- 能通过样本字段完整度、重复 URL 和结构指纹检查；
- 审计人显式批准后才可标记为“HTML 研究通过”。

HTML 研究通过不等于生产抓取批准。所有 HTML 候选在本批次继续保持 `manual_only` 或 `catalog_only`，不得被 Worker 作为 RawItem 抓取方式消费。若后续确实需要生产 HTML 抓取，必须单独设计 Html Ingestion 阶段和明确的批准字段，不能复用含义模糊的 `requires_manual_approval` 布尔值。

### 4. 有界验证与升级规则

验证只针对经过归因与方式复核的单个 Target，并强制复用现有 `new_safe_probe_client` / `safe_get` 边界，不能使用会自动跟随跳转的生产抓取客户端。每次最多五条样本、每主机并发 1、20 秒超时、2 MB 响应上限、最多五次显式审核跳转、每轮最多一个备用方式。401/403/登录/挑战立即停止；429 保存 `Retry-After` 和最早复查时间后停止，不在同一操作中等待或循环重试；5xx 和网络错误只记录为临时失败。

只有满足现有试用规则的非 HTML Target 才能成为“可试用抓取”：公开直连、`ready`、非 HTML 自动方式、无硬阻断、最新探测成功、有样本、字段完整度至少 60%，并同时取得标题和规范 URL。此规则保持现有 `evaluate_trial_eligibility` 对 HTML 的拒绝，不在本批次放宽。

修复成功只表示可以进入受控试用；仍需后续稳定性验证才能进入长期抓取。失败仍保留原始分类、建议动作、备用方式和复查条件。

## 数据、接口与网页

YAML 继续是人工审核的来源真相，程序不得自动修改 YAML。数据库保存不可变探测历史与当前状态。本批次优先复用现有 `source_research_profiles`、`source_acquisition_candidates` 和 `source_acquisition_probe_runs`，不为相同事实创建第二套表。修复审计至少投影：目标 ID、原失败探测 ID、归因、候选方式、审核结论、证据 URL、执行时间、结果、中文原因和下一步动作。

新增只读命令/报告能力应满足：

- 列出本批动态失败范围及其归因；
- 可从失败清单创建多个彼此独立的单 Target 操作，但每个操作只能验证已批准候选，不能在同一任务中盲目遍历未知端点；
- 生成 `reports/source-failure-remediation.md`，按归因、方式、结论和阻塞条件汇总；
- 在现有中文来源页和目标详情显示“修复建议/最后结论”，但不暴露凭据和网络细节。

网页不会直接执行审核、修改 YAML 或绕开限制；所有写操作保留在 CLI/Worker 受控流程。报告中的证据 URL 只显示移除查询参数和片段后的公开地址。

## 可观测性与可靠性

每次候选验证均写入操作日志、探测历史和脱敏事件。每个 Target 使用独立操作，单一目标失败不得中断其余目标；Worker 继续使用现有心跳、取消和恢复机制。本批次的网络验证操作在队列层限制为全局并发 1，避免多个 Worker 绕过进程内的主机限流；未取得执行权的 Worker 立即释放任务，不阻塞等待。日志可定位到来源、方法、阶段与错误类别，但不会输出 URL 查询中的秘密、请求头、Cookie、代理或密钥。

本批次必须定义独立于通用 `MAX_ATTEMPTS=3` 的修复重试策略：认证/政策、429、端点变化、字段不完整和未知错误均为非自动重试；网络临时失败最多允许一次延迟后的新操作，并且不得在 Worker 内阻塞等待。401/403/挑战立即停止，429 只记录服务端要求，5xx/网络错误只允许后续人工或计划内低频复测。

## 测试与验收

- 失败归因对 401、403、404、429、5xx、超时、解析错误、字段缺失和未知错误给出稳定分类与中文说明；
- 401/403/挑战/登录路径不会尝试 HTML 回退、Cookie 或第二种访问绕过；
- HTML 候选无论研究结果如何，本批次都不能进入自动试用或生产 RawItem 抓取；
- HTML 候选缺少 robots、条款、人工审核、审核域跳转或字段完整度任一条件时不能标记为研究通过；
- 候选验证不能使用生产抓取客户端，跨域跳转未在审核域列表中时必须停止；
- 官方 RSS/API 备用方式可以被审计和有界验证；
- 单一来源失败不影响其他已批准来源；
- 多 Worker 同时运行时，来源修复网络操作仍最多只有一个执行实例；
- 401、403、429、端点变化和字段不完整不会被通用三次重试机制重复请求；
- 报告不包含凭据、Cookie、代理、授权头或数据库连接信息；
- 不可变批次清单中的每个失败 Target 都有最终分类和可读结论，即使后续最新探测结果已经变化；
- 成功升级的 Target 保留与原始失败记录及审核证据的关联。

## 完成定义

本批次完成时，用户能从中文网页和 Markdown 报告准确看到：哪些来源确实可修复、使用什么合规方式、哪些因平台政策被阻断、哪些仅能发现线索，以及每项结论由哪次探测和审核支持。来源数量增长不是唯一指标；错误分类准确、访问边界不被突破、结论可追溯才是完成条件。
