# 中文日报人工审核与编辑覆盖层设计

## 目标

在既有“手动中文日报 MVP”上增加可追溯的人工审核能力。审核者可以为日报草稿中的每个条目维护中文标题、中文文章概述、审核结论、中文审核建议和中文证据评价；归档日报展示最终内容但不可修改。

本功能服务于日报编辑，不重新抓取来源、不重跑事件、不调用 MiniMax，也不修改 `RawItem`、`Event`、`EventVersion` 或已有事件证据。

## 已有边界

现有 `daily_report_items.snapshot` 是生成日报时固定的事件展示快照；日报详情页只读该快照。草稿目前支持收录/排除、排序和归档，归档后只能创建修订版。原设计明确禁止编辑快照中的标题和摘要。

新功能必须保留这个边界：人工文字不能覆盖或伪造原始事件快照，`emerging` 事件不能被人工改写为 `confirmed`。

## 方案比较

### 方案 A：直接修改 `snapshot`

实现最小，但会改变生成时固定的原始证据展示，无法区分模型/规则输出和人工改写，也破坏已归档日报的可追溯性。因此不采用。

### 方案 B：在 `daily_report_items` 直接保存当前编辑字段

查询简单，但每次保存都会覆盖上一版文字，无法回答“何时为何作出此审核结论”。这不满足人工审核留痕要求，因此不采用。

### 方案 C：追加式人工审核覆盖层（采用）

新增独立审核记录表。每次保存创建一条不可变记录，详情查询选择当前日报条目的最新记录作为显示覆盖层；无记录时回退到原始 `snapshot`。这既保留原始事实，又保留每次人工判断。

## 数据模型

新增 `daily_report_item_editorial_reviews`：

- `id`：主键。
- `daily_report_item_id`：外键指向 `daily_report_items`，删除日报时级联删除。
- `revision`：同一日报条目从 1 开始递增，和 `daily_report_item_id` 联合唯一。
- `decision`：仅允许 `keep`、`needs_evidence`、`exclude`、`duplicate`。
- `zh_title`：人工中文标题，长度 1–240 个字符。
- `zh_summary`：人工中文概述，长度 1–4,000 个字符。
- `review_recommendation`：中文审核建议，长度 1–2,000 个字符。
- `evidence_assessment`：中文证据评价，长度 1–2,000 个字符。
- `created_at`：服务端 UTC 时间。
- `copied_from_editorial_review_id`：可空自引用/跨条目引用，记录修订版从哪条最终审核记录复制而来。

所有文本为纯文本；服务端去除首尾空白、拒绝空值和超长值。记录不保存 Cookie、凭据、模型原始响应、外部网页正文或任何敏感配置。

`daily_report_items.snapshot` 保持不变。`included` 继续表示这条内容是否收录到本版日报；保存审核记录时按结论同步：

- `keep`、`needs_evidence`：`included=true`；
- `exclude`、`duplicate`：`included=false`。

这样“审核结论”和“是否收录”不会产生矛盾。现有快速收录/排除接口保留兼容性，但网页改为统一使用审核表单；兼容接口调用也必须追加一条无正文变更的审核操作记录，明确说明收录状态由快速操作变更。

## 读取与归档规则

详情查询对每条日报项目读取最新人工审核记录：

- 有审核记录：页面显示其中文标题、概述、结论、审核建议和证据评价，并标记“人工审核版本”。原始快照内容提供“查看生成原文”对照区域。
- 无审核记录：页面继续显示 `snapshot` 的标题、摘要和证据，明确标记“尚未人工审核”。
- 原始证据列表、事件编号、事件版本、section、确认状态、独立证据根数和限制说明始终来自固定快照，人工编辑不能修改。

`emerging` 条目的“尚未确认”标签始终展示；审核结论为 `keep` 也不能将它移动到“确认要闻”。

归档后的日报展示最终最新审核记录及完整历史列表，只读。创建修订版时，系统把父日报每个条目的最新人工审核记录复制为新条目的第一条审核记录，并写入 `copied_from_editorial_review_id`；父日报和其历史记录不变。

## 草稿编辑流程

日报草稿的每个条目提供“编辑中文审核内容”表单，预填当前显示内容：

1. 选择审核结论；
2. 填写中文标题与中文文章概述；
3. 填写中文审核建议与中文证据评价；
4. 提交后，服务端在短事务中验证日报和条目归属、确认仍为草稿、创建下一修订号的审核记录，并同步 `included`；
5. 重定向回日报详情页。

保存审核内容不会访问网络、不会调用模型、不会改变事件数据。单条验证失败仅返回该表单的中文错误，不影响日报其他条目。

## Web 接口与安全

新增 `POST /daily-reports/{report_id}/items/{item_id}/editorial-reviews`。它复用现有本机环回限制、同源检查、一次性动作令牌和中文错误映射。

路由只解析表单、调用日报 repository、重定向；字段验证与草稿状态检查在 repository/schema 层完成。条目必须属于请求日报；归档日报、新旧 URL 混用、非法结论、空文本、超长文本和并发修订冲突必须返回不泄漏内部代码或数据库信息的中文诊断。

展示文本采用模板默认转义，不渲染 HTML。公开证据 URL 继续走现有 `daily_report_public_url` 清洗，不由人工表单写入。

## 组件边界

- `newsradar.db.models`：新增审核记录 ORM 模型和约束。
- 新迁移：创建审核表、索引、外键和唯一约束，支持 PostgreSQL 升降级。
- `newsradar.daily_reports.schema`：审核结论常量、输入值对象和纯文本验证。
- `newsradar.daily_reports.repository`：追加审核记录、选择最新记录、修订版复制和草稿锁定。
- `newsradar.web.daily_report_queries`：将当前审核记录及审核历史投影到只读视图。
- `newsradar.web.app`：新增薄 POST 路由和中文错误映射。
- `daily_report_detail.html`：显示人工审核内容、原始快照对照、审核历史和草稿编辑表单。

不新增抓取器、Worker 任务、模型调用、权限绕过、定时任务、推送或推荐逻辑。

## 测试与验收

测试先行覆盖：

- schema 对四种结论、空白和长度边界的验证；
- repository 追加修订、不改原始快照、同步 `included`、归档锁定、条目归属和并发冲突；
- 修订版复制最新人工审核记录并保留原记录；
- 查询在有/无人工审核记录时的显示回退与历史排序；
- Web 表单的中文渲染、同源/动作令牌保护、错误中文化和 `emerging` 不可升级；
- PostgreSQL 迁移升级/降级；
- 真实本地网页：为当前草稿写入审核内容、确认三条保留项和两条排除理由均可见，且日报仍为草稿；归档验证仅在用户再次明确确认后进行。

最终验收运行完整 `pytest`、`ruff check .`、迁移测试和真实网页检查。网络抓取、模型调用、来源数量和已有用户报告均不在本功能修改范围内。
