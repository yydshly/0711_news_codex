# 来源重复占位项收口设计

## 1. 背景与结论

当前来源目录共有 166 个 Target，其中存在 19 组由早期批量建目录生成的
`universe-<provider>-1` 与 `universe-<provider>-2`。每组两个记录使用完全相同的访问地址，
但名称分别暗示“主入口”和“发现查询”，容易让用户误以为系统掌握了两个不同入口。

这 38 条记录均未启用，不影响当前 54 个已启用来源的抓取。本次只修正目录事实和中文展示，
不新增抓取器、不启用这些来源，也不修改 RawItem、事件、摘要或推荐流程。

## 3. 数据处理规则

每组统一按以下规则收口：

1. `universe-<provider>-1` 保留在目录中，研究状态改为 `duplicate`。
2. 重复项的中文结论明确写出其对应的规范记录 `universe-<provider>-2`，并说明不会参与抓取。
3. `universe-<provider>-2` 保留为唯一的间接新闻发现查询，研究状态改为
   `needs_research`。
4. 规范记录补充用途、所需信息、研究结论、风险结论和审核日期；本次不将其标记为
   `verified`，也不启用 ingestion。
5. 所有现有 ID 保持不变，避免破坏数据库版本快照、探测历史和页面链接。

处理后的预期变化：

| 指标 | 处理前 | 处理后 |
| --- | ---: | ---: |
| 本次涉及的占位项 | 38 | 0 |
| 明确保留的重复历史项 | 0 | 19 |
| 唯一、待研究的间接查询 | 0 | 19 |
| 本次新增启用来源 | 0 | 0 |
| 来源目录总数 | 166 | 166 |

## 4. 审计逻辑

来源审计不得再仅凭 `universe-*-1/2` 的 ID 命名推断占位状态。是否为占位、重复、
待研究或已验证，必须以 YAML 中的 `research.status` 为准。

审计保留以下检查：

- `research.status == placeholder` 时产生占位警告；
- 同一 Provider 下相同 `official_identity_url` 仍作为潜在重复提示；
- `needs_research` 不计为已验证，但可作为真实、待研究 Target 进入目录统计；
- `duplicate` 不计入真实覆盖和可抓取覆盖。

## 5. 网页展示

来源研究列表继续展示研究状态。Target 详情页需要用中文解释：

- `重复`：历史目录项，已被另一条规范记录替代，不会抓取；
- `待研究`：入口已明确，但尚未完成样本、字段、条款和备用方式验证；
- `占位`：只有平台或概念登记，尚未形成真实可探测 Target。

重复项的研究结论必须包含规范记录 ID，使用户能够搜索并打开对应记录。规范记录的结论必须
说明它是 Google News 或其他聚合方式的间接发现信号，不能作为事实证据。

## 6. 同步和运行边界

- YAML 仍是来源目录真相；通过现有 `sources sync` 幂等同步到 PostgreSQL。
- 同步只更新这 38 个现有记录，不删除定义和历史版本。
- 不执行网络探测或真实抓取，不影响运行中的 Worker。
- 同步后重启 Web，使研究状态与中文结论可见。
- 当前 54 个已启用来源及其 ingestion 配置必须保持不变。

## 7. 测试与验收

自动化测试必须证明：

1. 19 个 `*-1` 全部是 `duplicate`，且 ingestion 未启用；
2. 19 个 `*-2` 全部是 `needs_research`，且 ingestion 未启用；
3. 每组两个记录仍保持相同历史入口，但只有 `*-2` 被视为规范待研究查询；
4. ID 命名本身不再触发占位警告，真实 `placeholder` 状态仍会触发警告；
5. 目录总数保持 166，当前已启用来源保持 54；
6. 来源研究网页能显示“重复”“待研究”以及对应中文结论；
7. 完整 pytest 与 Ruff 检查通过；
8. PostgreSQL 同步后，网页中的 38 条记录呈现为 19 条重复历史项和 19 条待研究入口。

## 8. 非目标

- 不研究或启用这 19 个间接查询；
- 不处理其余平台占位项；
- 不删除数据库历史；
- 不开发新的 RSS、HTML、API 或第三方库抓取方式；
- 不修改事件、摘要、推荐、推送和调度能力。
