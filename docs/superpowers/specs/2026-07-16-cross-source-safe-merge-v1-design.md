# News Codex 跨来源安全合并 v1 设计

日期：2026-07-16
状态：设计已在对话中分段确认，待书面规格复核
适用分支：`codex/cross-source-safe-merge-v1`

## 1. 背景与真实基线

本阶段处理已经抓取并写入数据库的新闻，不新增来源、不重新探测来源，也不把聚类等同于删除原始文章。每篇原始内容继续作为独立 `RawItem` 永久保留；系统只判断哪些不同报道描述同一新闻事件，并把它们组织到同一个 `Event` 中。

2026-07-16 对当前代码与本地 PostgreSQL 的只读审查确认：

- 项目已有 `RawItem → EventCandidate → Event → EventVersion` 管线，不需要重建事件系统。
- 已有 `cluster-v3` 的有界候选生成、规则判定、MiniMax 边界辅助、持久化 pair decision、Worker 操作、事件租约、合并、拆分和重新聚类能力。
- 已有至少 50 个正例和 50 个反例的聚类回归集；既定目标为正例召回率不低于 85%，反例零误合并。
- 数据库共有 389 个事件，其中 315 个 `current`、74 个 `legacy`；状态为 6 个 `confirmed`、383 个 `emerging`。
- 315 个当前事件中，296 个只有一个成员，19 个有两个及以上成员；只有 18 个事件包含两个及以上来源。
- 31 条 RawItem 同时属于多个 `current` 事件；29 对当前事件存在活跃成员重叠，其中 28 对成员集合完全相同，主要由 `cluster-v2` 与 `cluster-v3` 候选身份变化产生。
- 最新成功的普通事件管线 Operation #781 固定的是 `cluster-v2` 快照，包含 86 个事件，其中 9 个跨来源事件；该不可变快照内部没有 RawItem 重复归属。
- 最新高价值 `cluster-v3` 波次 Operation #971 包含 36 个事件、38 条唯一 RawItem，仅有两对相同 Google News 链接的直接合并；快照内部没有 RawItem 重复归属。
- 全库 `cluster-v3` pair decision 为 2 个 merge、445 个 separate；两个 merge 都由 `same_canonical_url` 触发，没有模型自动合并。
- 实体提取把 `source_topics`、发布渠道和媒体名称混入新闻正文。Google News 条目因此普遍出现错误的 `organization:google`，造成大量无效候选和一次明显错误的模型边界候选。
- `/events/merge` 后端路由已经存在，但只校验两个正整数事件 ID；事件详情页没有合并、拆分入口，现有合并逻辑也没有对象、动作、时间、版本指纹或证据一致性的内容安全门。
- RawItem 重复候选页已有人工裁决，但它解决的是“是否为同一篇内容”，不能复用为“是否为同一新闻事件”。

因此，本阶段不是重新设计聚类器，也不以降低单成员比例为独立目标。工作重点是：消除确定的并行当前身份、修复实体输入污染、建立可审计的事件合并候选与人工确认闭环，并把现有人工 merge 收紧到安全执行路径。

## 2. 核心原则

采用“自动合并零误判优先；证据不足时人工确认”的保守原则：

1. 宁可暂时漏合并，也不自动误合并不同新闻。
2. 规则和数据库事实决定是否合法执行；MiniMax 只能生成受限解释，不能批准合并、改变来源合规结论或绕过本地门槛。
3. RawItem、旧事件、历史 EventVersion、证据、Operation 和模型审计记录均不删除。
4. 历史算法身份整理与内容层面的跨来源合并是两种不同操作，必须在数据、页面和日志中明确区分。
5. 所有写操作只由持久 Worker 执行；网页只入队，不直接改事件或调用外部网络。
6. 每次执行都绑定不可变事件版本和输入指纹；事件变化后旧决定不得静默重放。
7. 已归档中文日报继续引用原 EventVersion 快照，不受后续事件整理影响。

## 3. 目标与非目标

### 3.1 目标

1. 修复事件实体提取的来源主题与采集渠道污染。
2. 识别并安全退役由算法身份变化产生的完全重复 `current` 事件。
3. 为确定性强身份建立安全自动合并路径。
4. 为疑似同一新闻建立中文、可解释、可过期的人工确认候选。
5. 合并后重新计算成员、证据、状态、评分和展示层级，并生成新 EventVersion。
6. 在网页汇总事件成员质量、重复当前身份、自动可处理和待人工处理数量。
7. 保留完整审计链，并支持通过事件版本和现有拆分能力纠错。

### 3.2 非目标

- 不新增 Provider、Target、RSS、API、Sitemap 或 HTML 抓取器。
- 不重新抓取或重新探测来源。
- 不删除或物理合并 RawItem。
- 不让模型全局搜索、自由聚类或自动批准合并。
- 不因追求固定合并数量而放宽安全阈值。
- 不修改来源合法性、启用状态、凭据或付款结论。
- 不开发推荐、推送、定时日报或复杂摘要。
- 不回写或改变已归档日报。

## 4. 方案比较与选择

### 4.1 方案 A：只放宽现有聚类规则

实现最少，但无法解决 v2/v3 并行当前身份、实体污染和人工审计缺口。放宽相似度还会直接破坏零误合并目标，不采用。

### 4.2 方案 B：三层安全收口（采用）

将问题拆分为三个互不混淆的通道：

1. 历史身份整理：完全相同成员集合的旧算法事件退役为 `legacy`。
2. 确定性内容合并：只接受不可变的同一内容或同一官方事实标识。
3. 人工确认：对象、动作、时间相近但证据不足的候选只展示，不自动修改事件。

该方案复用现有 EventVersion、Operation、Worker、租约和网页架构，风险最低且可审计。

### 4.3 方案 C：模型判断并自动合并

可能提高召回，但不可稳定解释，且会把模型升级成事实裁决者，不采用。

## 5. 概念与数据边界

### 5.1 RawItem 重复与 Event 合并

- RawItem 重复：两条采集记录是否指向同一篇原始内容。现有 `duplicate_candidates` 继续负责该语义。
- Event 合并：多篇不同报道是否描述同一新闻事件。新增独立事件合并候选表，不复用 RawItem 重复候选。
- 历史身份重复：同一成员集合被不同聚类算法候选键发布为多个 `current` Event。这是事件身份收口，不是新闻内容合并。

### 5.2 新增事件合并候选

新增持久记录 `event_merge_candidates`，至少包含：

- `id`
- `left_event_id`、`left_version_number`
- `right_event_id`、`right_version_number`
- `candidate_type`：`legacy_identity`、`deterministic_merge`、`manual_review`
- `status`：`pending`、`confirmed`、`dismissed`、`applied`、`expired`、`failed`
- `algorithm_version`
- `input_fingerprint`
- `facts_snapshot`：有界、清洗后的成员 ID、来源 ID、发布者、发布时间、规范化 URL 身份、对象、动作、证据根和冲突标记
- `reason_codes`
- `zh_reason`、`zh_next_action`
- `generated_operation_id`
- `reviewed_operation_id`、`applied_operation_id`
- `reviewed_at`、`created_at`、`updated_at`

表级约束必须保证：

- 左右 Event ID 按升序规范化且不相同。
- 事件版本号均为正整数。
- 同一算法、左右版本和输入指纹最多一个候选。
- 状态与审核/执行字段保持一致。
- `facts_snapshot` 不保存完整 payload、Cookie、Authorization、API Key、数据库连接串、完整模型提示或含敏感查询参数的 URL。

### 5.3 输入指纹与过期

候选输入指纹必须覆盖：

- 两个 Event ID 与当前版本号；
- 各自活跃 RawItem ID 集合；
- 每个成员的安全 URL 身份、发布时间、规范化标题摘要、对象与动作；
- 两侧证据根摘要；
- 候选算法版本。

执行前重新计算指纹。任一事件版本、成员、URL 身份或关键事实发生变化时，候选转为 `expired`，不执行原决定。

## 6. 实体提取修复

### 6.1 根因

`extract_entities()` 当前读取标题、摘要、正文、item kind、publisher 和 `source_topics`。Google News 的来源主题或渠道标识含 Google，导致几乎所有条目被错误打上 `organization:google`；发布媒体名也可能被误当成新闻主体。

### 6.2 新边界

- 事件实体只从标题、摘要和有界正文提取。
- `source_topics` 继续服务相关性判断，但不参与事件实体识别。
- `publisher_name`、采集平台和来源入口只用于证据归属，不自动成为新闻主体。
- 只有新闻文本真实提到 Google、Reuters 等实体时才识别。
- 升级实体算法版本，保留旧处理记录和历史 EventVersion，不覆盖历史快照。
- 重新处理只发生在新的候选生成 Operation 内，不通过直接数据库脚本静默改历史记录。

## 7. 候选生成通道

### 7.1 历史身份整理

仅在以下条件全部成立时生成可自动应用的 `legacy_identity` 候选：

1. 两个事件均为 `current` 且有完整当前 EventVersion。
2. 两个事件的活跃 RawItem 集合完全相同且非空。
3. 一个候选身份明确来自旧聚类算法，另一个来自当前聚类算法。
4. 最新适用 Operation 快照明确引用当前算法事件。
5. 两个事件当前都没有未过期的编辑租约。
6. 没有已确认的人工决定要求保留旧身份。

成员集合仅为子集、部分交叠或存在不同 nightly/release 标识时不得自动退役，转为人工复核或保持分开。

应用时保留当前算法事件不变，将旧算法事件写入一个新的审计版本并把 `visibility` 改为 `legacy`。不得把旧事件标记为 `rejected`，因为新闻事实没有被否定。

### 7.2 确定性内容合并

只有以下强身份可形成自动可应用的 `deterministic_merge` 候选：

- 相同的真实原始媒体 URL；
- 相同 Canonical URL；
- 相同仓库 release、tag 或 commit 稳定标识；
- 相同 DOI、arXiv ID 或其他论文稳定标识；
- 相同官方公告的稳定原始标识。

强身份必须经过来源类型和 URL 语义校验。Google News 中间 URL、聚合页 URL、仅标题指纹相同、仅发布者相同、仅公司相同或模型高置信度都不能单独触发自动合并。

### 7.3 人工确认候选

只有同时满足以下条件时才生成 `manual_review`：

- 两侧存在相同的具体对象实体，而非泛化 AI 词或仅组织名；
- 动作一致；
- 发布时间在确定的候选窗口内；
- 没有互斥对象、动作、版本号、金额或其他关键数字；
- 至少来自两个不同采集入口；
- 两侧事件当前没有活跃成员重叠，或者重叠关系被明确标记为历史身份问题。

候选默认不修改事件。标题相似度和模型意见只能解释优先级，不能批准执行。

### 7.4 明确保持分开

以下情况直接记录稳定原因并保持分开：

- 具体对象互斥；
- 动作冲突；
- 关键版本、金额、主体或日期冲突；
- 仅共享组织名或通用 AI 术语；
- 超出时间窗口且没有强身份；
- 只有聚合渠道身份相同；
- 输入不足以验证原始媒体归属。

## 8. 执行语义

### 8.1 网页与 Worker 边界

网页只允许：

- 入队生成/重新检查候选 Operation；
- 入队应用确定性候选 Operation；
- 提交人工 `确认合并` 或 `保持分开` 决定并入队执行。

网页请求中不得运行聚类、调用 MiniMax、获取事件租约或直接写 Event/EventVersion。

### 8.2 执行前重验

Worker 执行前必须重新确认：

- 候选仍为允许执行的状态；
- 左右 Event 仍存在且为 `current`；
- 当前版本号与候选快照一致；
- 活跃成员和输入指纹一致；
- 候选类型与审核决定匹配；
- 没有内容冲突或新增安全阻断；
- 两个事件可以按 ID 升序取得租约。

任一条件失败时不得部分修改事件；候选标记 `expired` 或 `failed`，并记录稳定错误码、中文原因与下一步动作。

### 8.3 内容合并

合并成功时：

1. 按确定性规则选择 survivor；不得仅以用户提交顺序决定。
2. 合并双方 RawItem ID 集合，不复制、不删除 RawItem。
3. 从持久事实重新构造候选，重新计算证据、状态、评分、层级和中文回退内容。
4. survivor 创建新的 EventVersion 并原子切换当前版本。
5. 被合并事件创建空成员审计版本并转为 `legacy`；不得使用 `rejected` 表示已合并。
6. 候选转为 `applied`，记录 Operation 与产生的版本号。
7. 释放租约；任一写入失败必须整体回滚，不遗留租约或半合并状态。

### 8.4 并发、重试与批次隔离

- 两个事件始终按 ID 升序加租约，逆序释放。
- Operation 使用现有截止时间、取消、心跳、有限重试和过期租约恢复能力。
- 重试必须复用同一候选 ID 与输入指纹，保证幂等。
- 单个候选失败只记录该候选结果，不阻塞同批其他候选。
- 网络或模型调用不得持有数据库事务或事件租约。
- 候选安全判断不依赖模型，因此模型不可用时整批仍可完成。

## 9. 中文网页

新增“事件合并候选”页面，并复用现有导航、Operation 页面和事件详情页样式。

### 9.1 汇总指标

- 当前事件总数；
- 单条目事件数；
- 跨来源事件数；
- RawItem 同时属于多个 current Event 的数量；
- 历史身份重复候选数；
- 确定性自动候选数；
- 等待人工确认数；
- 已应用、已排除、已过期和失败数。

### 9.2 候选详情

每个候选必须展示：

- 两侧 Event ID、版本号、中文标题与当前可见性；
- 原始报道标题、媒体、来源入口和发布时间；
- 去除查询参数与凭据后的安全 URL；
- 活跃 RawItem 成员；
- 共同对象、动作、时间差、关键数字冲突；
- 独立证据根与聚合转载关系；
- 候选类型和稳定原因代码；
- 中文结论：为什么可能相同、为什么不能自动合并；
- 中文下一步动作。

历史算法重复必须显示“旧算法身份重复”，不能冒充内容层面的跨来源报道合并。

### 9.3 人工操作

- `确认合并`
- `保持分开`
- `重新检查`
- `查看两个事件详情`

确认按钮只对仍有效的人工候选开放。自动候选也应先可见、可抽查，再由受控 Operation 批量或单项应用。所有 POST 继续使用本机来源校验和一次性操作令牌。

## 10. 中文诊断与安全

稳定错误码至少覆盖：

- `event_merge_candidate_expired`
- `event_merge_version_changed`
- `event_merge_membership_changed`
- `event_merge_identity_not_strong`
- `event_merge_fact_conflict`
- `event_merge_lease_unavailable`
- `event_merge_quality_input_unavailable`
- `event_merge_already_applied`
- `event_merge_candidate_failed`

页面把错误码投影为中文原因和下一步动作，不输出原始异常、SQL、连接串、Key、Cookie、Authorization、完整模型提示或带敏感参数的 URL。Operation 日志只记录有界 ID、阶段、错误码和安全摘要。

## 11. 与现有能力的关系

- 继续复用 `EventRepository`、`EventPublisher`、EventVersion、事件租约和 Operation Worker。
- 继续复用 `cluster-v3` 的有界候选思想，但事件间候选生成与 RawItem pair decision 分开持久化。
- 现有 `/events/merge` 不再允许绕过候选安全门；兼容入口必须要求有效候选 ID 或被替换为候选执行路由。
- 现有 split/recluster 继续作为纠错能力，但事件详情页只在合适上下文提供受控入口。
- RawItem `duplicate_candidates` 保持原语义，不自动驱动 Event 合并。
- 日报只读取固定 EventVersion；归档日报的触发器和不可变性不作修改。

## 12. 测试设计

### 12.1 实体提取

- Google News 的 `source_topics` 或 publisher 不再产生 `organization:google`。
- 新闻标题/摘要/正文真实出现 Google 时仍能识别。
- Reuters、WSJ 等媒体名不会仅因 publisher 字段成为事件主体。
- 旧实体处理记录与历史 EventVersion 不被覆盖。

### 12.2 候选规则

- v2/v3 活跃成员集合完全相同且最新快照引用 v3 时，生成可退役旧身份候选。
- 成员为子集、部分重叠或不同 nightly release 时禁止自动退役。
- 相同真实原始 URL、repository release ID、DOI/arXiv ID 可形成确定性候选。
- Google News 中间 URL、仅标题相似、仅公司相同、仅时间接近时不得自动合并。
- 对象、动作、版本或关键数字冲突时必须保持分开。
- 模型高置信度且无强身份时仍不能形成自动候选。
- 输入变化后候选变为过期。

### 12.3 执行与并发

- 合并重新计算证据、状态、评分和层级。
- survivor 产生新版本，被合并事件产生空成员审计版本并转为 `legacy`。
- 事件版本变化后旧候选不能执行。
- 并发 Worker 按 ID 顺序加锁，不重复合并、不遗留租约。
- 截止时间、取消和失败整体回滚。
- 单个候选失败不阻塞整批。
- 重试幂等，不重复生成版本。
- 归档日报快照完全不变。

### 12.4 网页与安全

- 页面汇总和候选详情使用中文、稳定原因和明确下一步。
- 历史身份整理与内容合并标签不可混淆。
- 无效、过期或已应用候选不能重复提交。
- CSRF/本机来源保护继续生效。
- HTML、日志和候选快照不泄露敏感查询参数、凭据、连接串或原始异常。

### 12.5 现有回归

- 聚类标注集正例召回率不低于 85%。
- 反例零误合并。
- 完整 pytest 与 Ruff 通过。
- 既有抓取、Worker、事件、证据、网页和日报测试不回归。

## 13. 真实 PostgreSQL 与浏览器验收

1. 先只读生成候选，不立即修改事件。
2. 记录生成前事件、版本、成员、Operation、RawItem、日报和模型调用计数。

## 14. 完成条件

第一阶段不追求固定合并数量，完成条件为：

> 消除确定的重复当前身份，修复实体污染，建立可审计的安全候选和人工确认闭环，同时保持自动误合并为零。

量化验收：

- 最新候选生成范围内，RawItem 不再同时属于多个 current Event；历史目录中的可解释残留单独列出。
- 所有自动应用候选均有强身份或严格的完全重复身份依据，人工抽查零误处理。
- 聚类回归集继续满足正例召回率不低于 85%、反例零误合并。
- 所有人工候选都有唯一中文结论、原因和下一步动作。
- 事件合并不删除 RawItem、EventVersion、证据、Operation 或已归档日报。
- MiniMax 完全不可用时，候选生成、执行安全与中文规则诊断仍能完成。
- 全量 pytest、Ruff、真实 PostgreSQL、真实浏览器和 8766 健康检查通过。

## 15. 实施顺序建议

后续实施计划应按以下大里程碑组织，并采用 TDD：

1. 实体输入边界与算法版本修复。
2. 事件合并候选 schema、迁移、repository 和只读生成器。
3. 历史身份整理与确定性/人工候选规则。
4. Worker 安全执行、版本重验、合并重算和批次隔离。
5. 中文网页、人工裁决和诊断投影。
6. 真实数据库候选抽查、受控应用、浏览器和全量验收。

任何里程碑都不得通过直接脚本修改真实事件；真实写入只在前序测试和候选人工抽查通过后，由正式 Worker Operation 执行。
