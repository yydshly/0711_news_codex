# News Codex 中文来源感知台设计规格

## 1. 目标

为 News Codex 增加一个本机运行、中文优先、只读的网页版来源感知台。它不展示新闻摘要，而是帮助使用者理解当前来源系统的真实能力：已经登记哪些平台和目标、哪些可以直接读取、哪些只能间接发现、哪些被凭据/审批/付费阻塞、最近探测是否健康，以及每种来源采用什么方式获取。

首版以方案 A“总览指挥台”为主要视觉和信息骨架，同时融合：

- 方案 C 的中文诊断叙事，用自然语言回答“能看到什么、缺什么、为什么、下一步是什么”；
- 方案 B 的来源下钻能力，让 Provider、Target、访问方式、风险和探测历史可以逐层查看。

## 2. 范围与约束

### 2.1 包含

- 本机网页服务，仅监听 `127.0.0.1`，默认端口 `8765`。
- 从现有 PostgreSQL 读取 Provider、Target、风险和探测记录。
- 中文界面与中文解释；Provider、协议、枚举和技术标识保留英文原值作为辅助信息。
- 总览、来源能力、探测记录、目标目录、阻塞与解锁五个只读视图。
- Provider 和 Target 的详情抽屉或详情页。
- 按来源类别、可用性、覆盖方式、Provider、探测结果进行筛选。
- 数据库不可用、尚未同步、尚未探测时的明确空状态。
- 通过 CLI 启动网页服务。

### 2.2 不包含

- 不修改 YAML、数据库来源定义或状态。
- 不从网页触发同步、探测、抓取或 MiniMax 调用。
- 不做正式新闻抓取、摘要、推荐、推送和定时任务。
- 不做用户系统、鉴权、互联网部署或移动端原生应用。
- 不展示或记录 `.env`、API Key、Cookie、请求认证头等秘密信息。
- 不把 Provider 能力探测成功描述为内容已经覆盖。

## 3. 技术方案

采用同一 Python 工程内的轻量服务端渲染方案：

- FastAPI 提供本地 HTTP 服务与只读路由；
- Jinja2 生成 HTML；
- 原生 CSS 和少量原生 JavaScript 实现布局、筛选、抽屉和响应式交互；
- SQLAlchemy 2 复用现有数据库会话和模型；
- Typer 新增 `newsradar web` 启动命令；
- Uvicorn 作为本机开发运行器。

不引入 Node.js 前端工程。当前界面以读取、解释和下钻为主，服务端渲染能复用 Python 数据模型，减少重复 API 类型和构建链，也符合“本机、只读、先理解能力”的范围。页面必须在关闭 JavaScript 时仍能完成基本导航和阅读；JavaScript 只增强筛选和抽屉体验。

## 4. 信息架构

### 4.1 总览指挥台 `/`

首屏回答四个问题：

1. 当前登记了多少 Provider 和 Target？
2. 其中多少可以免费直接读取、多少只能间接发现、多少被阻塞？
3. 最近的真实探测健康度如何？
4. 最大的覆盖缺口和下一步解锁建议是什么？

页面区域：

- 左侧固定导航：总览指挥台、来源能力、探测记录、目标目录、阻塞与解锁。
- 顶部状态栏：数据库连接状态、数据最后更新时间、当前为“只读本机模式”。
- 关键指标卡：Provider 总数、Target 总数、免费直接覆盖、间接覆盖、阻塞目标、三轮连续成功数。
- 七类来源分布：专业媒体、官方一手、社交社区、聚合搜索、研究开发、通讯播客、趋势商业。
- 中文能力诊断：三段固定结构——“当前能感知”“主要盲区”“建议下一步”。内容由确定性规则生成，不调用 MiniMax。
- 最近探测：展示最近探测结果、延迟、字段完整度和失败原因。
- 重点阻塞：按影响与可解锁性列出 X、Facebook、Instagram、TikTok、LinkedIn 等受限平台。

### 4.2 来源能力 `/providers`

以 Provider 为行显示：平台名称、类别、成本、认证方式、可用性、能力、Target 数量、直接/间接覆盖数、最近能力探测结果、审核时间。

支持类别、可用性、成本和文本搜索。点击 Provider 进入 `/providers/{provider_id}`，展示：

- 中文性质说明；
- 官方主页、文档、条款和证据链接；
- 所需环境变量名称，但不显示其值；
- 解锁条件；
- 关联 Target；
- 最近能力探测记录；
- 明确区分“能力已确认”和“内容已读取”。

### 4.3 探测记录 `/probes`

统一呈现内容探测和 Provider 能力探测，但用明显标签区分二者。默认按时间倒序，显示对象、探测类型、结果、HTTP 状态、延迟、完整度、建议状态和中文原因。

支持结果、探测类型、Provider 和时间范围筛选。失败记录不得只显示错误码，必须使用规则映射给出中文含义，例如“需要凭据”“触发限流”“远端结构可能变化”“网络超时”。保留原始英文 reason 供展开查看。

### 4.4 目标目录 `/targets`

展示具体账号、频道、媒体栏目、社区、查询和信号。每行包含 Target 名称、Provider、目标类型、用途角色、覆盖方式、可用性、访问协议、风险分、最近内容时间和最新探测结果。

点击 Target 进入 `/targets/{source_id}`，展示 YAML 审核信息、首选与备用访问方式、预期字段、最近样本字段完整度、风险分项、连续探测情况和关联证据。访问 URL 可以显示；认证头、Cookie 和环境变量值必须隐藏。

### 4.5 阻塞与解锁 `/gaps`

按 `requires_credentials`、`requires_approval`、`requires_payment`、`manual_only`、`unavailable` 分组。每项展示影响范围、当前替代路径、预计成本级别、解锁步骤和证据链接。

页面必须明确指出：受限平台仍被来源地图覆盖，但当前不能等同于实时内容覆盖；系统不会使用 Cookie、验证码绕过或未经审核的网页爬取。

## 5. 中文表达规则

建立集中式枚举翻译表，不在模板中分散硬编码。至少覆盖 ProviderCategory、Availability、CoverageMode、TargetType、SourceNature、SourceRole、SourceStatus、AccessKind、探测 outcome、风险等级和成本等级。

中文文案遵循：

- “已登记”表示目录存在，不表示已抓取内容；
- “可直接读取”表示存在可用的合规访问路径；
- “间接发现”表示通过聚合、搜索或替代路径获得线索；
- “能力探测通过”只表示接口、文档或权限状态得到确认；
- “内容探测通过”才表示实际获得样本；
- 社交内容统一标记为“发现/热度信号，不作为独立事实证据”。

中文诊断由规则生成：统计覆盖和阻塞状态，识别前三个缺口，并根据可用性与成本给出确定性建议。首版不调用 MiniMax，避免模型把目录覆盖误写成内容覆盖。

## 6. 数据读取与派生

新增只读查询服务，将 ORM 记录转换为页面专用 ViewModel。模板不能直接执行数据库查询。

主要数据对象：

- `DashboardSummary`：总数、覆盖方式、可用性、类别、探测健康度和时间戳；
- `ProviderRow` / `ProviderDetail`：平台级能力、成本、阻塞和 Target 汇总；
- `TargetRow` / `TargetDetail`：具体目标、访问方式、风险和最新内容探测；
- `ProbeRow`：统一后的能力探测或内容探测记录；
- `GapGroup`：阻塞类别、影响数量、替代路径和解锁步骤；
- `DiagnosticNarrative`：能感知、主要盲区、建议下一步。

聚合口径：

- Provider 和 Target 总数来自当前定义表；
- “免费直接覆盖”要求 `coverage_mode=direct`、`availability=ready`，且 Provider 成本不是付费级；
- “阻塞目标”要求 `availability != ready`；
- 最新状态按每个对象最近一次探测记录计算；
- 三轮连续成功按每个 Target 最近三次内容探测全部成功计算；
- 没有探测记录时显示“尚未探测”，不得推断为成功或失败。

## 7. 视觉与交互

采用方案 A 的深色指挥台风格：深蓝黑背景、青绿色表示健康、蓝色表示信息、琥珀色表示受限、红色表示失败。内容区保持高对比度，正文不使用低于 14px 的字号。

桌面端为固定侧栏加主内容；窄屏下侧栏折叠为顶部导航，指标卡改为单列或双列。表格在窄屏使用水平滚动，不隐藏关键状态列。所有颜色状态同时配文字或图标标签，不能只靠颜色传达。

交互规则：

- 指标卡可点击并跳转到已应用对应筛选的列表；
- 所有筛选同步到 URL 查询参数，便于刷新和复制；
- 详情页提供返回列表链接；
- 外部证据链接在新标签页打开并标明外部链接；
- 不提供任何“启用”“编辑”“重新探测”按钮，避免只读界面造成错误预期。

## 8. 错误与空状态

- 数据库不可连接：返回可读的本地故障页，说明运行 `newsradar db start`，不得显示连接密码或堆栈。
- 数据表未迁移：说明运行 Alembic 升级命令。
- Provider/Target 尚未同步：显示同步命令和空状态，不伪造演示数据。
- 尚无探测历史：总览仍展示目录覆盖，并明确标记健康度“暂无数据”。
- 单条数据字段缺失：显示“未记录”，页面其余部分继续渲染。
- 未知 ID：返回中文 404 页面。

## 9. 安全边界

- 默认仅绑定 `127.0.0.1:8765`；CLI 不提供默认公网监听。
- 页面没有写数据库的路由和表单。
- 所有查询参数经过类型和枚举校验。
- 所有模板默认转义；数据库 reason、notes 和名称均视为不可信文本。
- 环境变量只判断“是否配置”，不把值传入 ViewModel、模板或日志。
- 响应增加基础安全头：`X-Content-Type-Options: nosniff`、`Referrer-Policy: no-referrer`、限制性的 `Content-Security-Policy`。

## 10. 测试与验收

### 10.1 自动化测试

- 查询服务对空数据库、完整数据和缺失探测历史生成正确统计。
- 免费直接覆盖、间接覆盖、阻塞和三轮连续成功口径准确。
- Provider 能力探测与 Target 内容探测不会混淆。
- 所有关键枚举都有中文标签，未知值安全回退到原值。
- 五个列表/详情路由返回 200，未知 ID 返回 404。
- 筛选查询参数生效，非法枚举返回可读的 422 或回退。
- HTML 不包含数据库密码、API Key、Cookie 或认证头。
- 数据库不可用时返回中文故障页。
- CLI 默认监听 `127.0.0.1:8765`。

### 10.2 浏览器验收

- 首页首屏能直接看到总量、直接覆盖、间接覆盖、阻塞和诊断结论。
- 点击指标能进入已筛选列表。
- 从 Provider 能下钻到关联 Target，从 Target 能查看探测与风险。
- X、Facebook、Instagram、TikTok、LinkedIn 均可在阻塞与解锁页找到。
- 页面在 1440px 和 390px 宽度下可用，键盘可以访问导航、筛选和链接。
- 关闭 JavaScript 后仍能导航和阅读主要内容。

## 11. 完成定义

运行 `uv run newsradar web` 后，使用者可以在本机中文网页中准确理解当前 67 个 Provider、161 个 Target 的登记覆盖、真实可读能力、探测健康度、访问方法、风险和解锁条件。页面不会把目录、能力探测或社交信号误述为已经获取并证实的新闻，也不会改变任何来源配置或运行状态。
