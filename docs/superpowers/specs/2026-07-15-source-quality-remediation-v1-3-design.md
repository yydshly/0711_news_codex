# 来源质量修复 v1.3 设计

## 1. 背景与目标

当前来源健康探测中有 10 个来源被标记为降级。它们并非同一种故障：部分来源能够返回内容，但协议字段没有完整映射；部分接口正常却暂时没有内容；另有来源入口本身不适合作为稳定内容入口。现有报告把这些情况统一显示为“降级”，用户难以判断应该修代码、等待内容，还是更换入口。

本轮目标是精准修复这 10 个来源的探测质量，让系统能明确区分：

- 内容获取成功且字段达到质量要求；
- 入口正常但当前没有内容；
- 内容存在但字段映射不完整；
- 访问失败、认证阻塞或入口失效。

本轮不新增摘要、推荐、事件聚合、定时调度或高风险网页抓取能力，也不降低全局 90% 字段完整率标准。

## 2. 范围

本轮只处理当前已确认的 10 个降级目标：

- Bluesky：`anthropic-bluesky`；
- RSS：`deepmind-blog`、`huggingface-blog`、`universe-latent-space-2`；
- Mastodon：`mastodon-ai-tag`、`mastodon-artificialintelligence-tag`、`mastodon-llm-tag`、`mastodon-machinelearning-tag`、`mastodon-mastodon`；
- GitHub Releases：`qwen3-releases`。

同时修复上述来源 YAML 中已经出现的中文乱码，避免网页继续展示不可读文本。除此之外，不进行来源宇宙扩容和无关重构。

## 3. 方案选择

采用“按协议精准修复”的方案：

1. 在现有协议探测器内补齐平台原生字段到 `ProbeSample` 的映射；
2. 使用现有 `ProbeResult.error_code` 表达可机器识别的空内容和字段不足原因；
3. 保持 `ProbeOutcome` 现有枚举和数据库结构兼容；
4. 在网页查询层把错误码、样本数和完整率转换为明确的中文状态说明；
5. 仅在入口确有问题时调整对应 YAML，不通过降低预期字段或全局阈值美化结果。

不采用全局降低完整率阈值，因为这会把真实字段缺失误判为健康；不为每个来源创建独立抓取器，因为同协议来源应该共享稳定、可测试的适配逻辑。

## 4. 协议适配设计

### 4.1 Bluesky

`BlueskyProbe` 继续使用官方公共 AppView 接口。规范化字段包括帖子文本、作者、创建时间、帖子 URL，以及点赞、转发和回复形成的互动总量。帖子文本可同时作为标题候选和正文，但标题必须经过现有长度与空值规则处理。

若接口返回合法响应但 `feed` 为空，结果为 `degraded`，错误码为 `no_content`，原因显示“入口正常，当前没有可用帖子”。这不会被计为内容探测成功。

### 4.2 Mastodon

新增 Mastodon 专用规范化逻辑，识别状态对象中的：

- `url` 或 `uri`：规范链接；
- `created_at`：发布时间；
- `account.display_name`、`account.acct` 或 `account.username`：作者；
- `content`：去除 HTML 标签后的正文与标题候选；
- `replies_count`、`reblogs_count`、`favourites_count`：互动总量。

仅处理公开 API 返回的数据，不请求登录页面，不使用 Cookie，也不跨实例抓取整个站点。空数组使用 `no_content`；内容存在但仍达不到预期字段时使用 `incomplete_fields`。

### 4.3 RSS/Atom

保留 `feedparser`，补齐常见字段兼容：

- 摘要依次读取 `summary`、`description`，必要时从 `content[0].value` 提取；
- 内容依次读取 `content[0].value`、`summary`、`description`；
- 日期继续兼容 `published` 与 `updated`；
- HTML 文本只在内存中清理标签用于字段判断，不自动抓取文章页面。

如果 feed 合法但没有条目，使用 `no_content`，而不是模糊的 0% 字段完整度原因。

### 4.4 GitHub Releases 与间接搜索

`qwen3-releases` 的官方 Releases 接口当前可能返回 HTTP 200 和空数组。这代表“接口可达，但仓库尚未发布 Release”，不是网络故障。保持来源不能晋级为有效内容来源，并以 `no_content` 和明确中文原因展示。

`universe-latent-space-2` 必须优先确认其现有官方 feed 是否仍可作为直接入口。若官方 feed 可用，则将目标改为经过审核的直接 RSS；若不可用，则保留间接发现属性并将空搜索结果标为 `no_content`。不得用未经审核的 HTML 抓取作为自动回退。

## 5. 状态与数据兼容

本轮不新增数据库枚举或迁移。使用现有字段表达结果：

| 情况 | outcome | error_code | 是否可视为内容成功 |
|---|---|---|---|
| 样本存在且完整率至少 90% | `success` | 空 | 是 |
| 合法响应但无样本 | `degraded` | `no_content` | 否 |
| 样本存在但字段不足 | `degraded` | `incomplete_fields` | 否 |
| 缺少凭据或被限流 | `blocked` | 现有具体错误码 | 否 |
| 网络、协议或解析失败 | `failed` | 现有具体错误码 | 否 |

来源资格晋级仍要求连续三次真实 `success`。`no_content` 只能证明入口可达，不能替代内容覆盖证据。

## 6. 网页与报告展示

来源列表、来源详情和修复报告复用同一套中文诊断规则：

- `no_content`：入口正常，当前没有可用内容；
- `incomplete_fields`：已获取内容，但缺少必要字段；
- `blocked`：需要凭据、审批、额度或等待限流恢复；
- `failed`：网络、响应格式或入口失效；
- `success`：内容与必要字段均满足当前标准。

页面继续显示样本数、字段完整率、最近探测时间和原始原因，中文结论不能覆盖原始证据。来源 YAML 的中文名称、用途、结论和风险说明统一保存为 UTF-8 可读文本。

## 7. 错误处理与可靠性

- 单个来源失败不能中断批量探测；
- 所有请求继续使用现有超时、限流和敏感头过滤机制；
- 空响应必须显式分类，不能抛出无意义异常；
- HTML 清理只处理已经通过 RSS/API 获得的字段，不发起额外页面请求；
- 不记录密钥、Cookie 或 Authorization 内容；
- 不自动修改来源资格状态，YAML 仍是人工审核真相。

## 8. 测试与验收

所有行为修改使用测试先行：先增加能够复现问题的失败测试，再实现最小修复。

自动化验收包括：

- Bluesky 互动总量与空 feed 分类；
- Mastodon 作者、正文、链接、时间和互动量规范化；
- RSS 摘要/正文回退以及空 feed 分类；
- GitHub Releases 空数组分类；
- 中文诊断映射与原始证据保留；
- 批量探测中单源失败隔离；
- YAML 严格校验、UTF-8 文本和凭据泄漏检查；
- 全量单元测试、代码检查和差异检查。

真实网络验收对修复目标至少执行一轮定向探测；能够返回内容的来源再执行三轮连续探测。暂时无内容的来源以入口可达证据验收，不伪造三轮内容成功。最终生成中文来源质量报告，逐项列出修复前状态、修复后状态、剩余限制和下一步建议。

## 9. 完成条件

- 10 个来源均获得明确且可审计的分类；
- Mastodon、Bluesky 与 RSS 的字段映射有固定样本测试保护；
- 空内容不再与网络故障混淆，也不被计为成功覆盖；
- 网页能够用中文解释每个来源为何成功、降级或阻塞；
- 不降低全局质量门槛，不启用 HTML 自动回退或登录态抓取；
- 自动化测试通过，真实探测报告生成，代码可进入合并前审查。
