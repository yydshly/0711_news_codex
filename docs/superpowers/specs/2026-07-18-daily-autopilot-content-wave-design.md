# 自动日报真实抓取与完整生成设计

## 目标

让“开始自动日报”真正执行一轮有边界、可恢复的内容采集，并基于该批次的固定事件快照生成中文决策简报、情报全览和两份语音。单个来源失败不得阻塞整批；来源合法性和启用状态继续由现有确定性配置决定，MiniMax 只参与中文表达和语义辅助。

## 已确认的问题

当前自动日报执行的是：来源目录刷新、独立事件处理、日报、审核、归档和音频。来源目录刷新只运行内容探测、平台能力探测和目录判断，不创建 FetchRun，也不写入 RawItem。因此事件处理只能读取数据库中的旧数据。

生产数据库已经证明这不是来源本身无新闻：自动日报第二轮完成 187 个目录成员后，最近 24 小时仍为 0 个 FetchRun、0 个新增 RawItem；而最近一次高价值波次为 41 个成员创建了 34 个 FetchRun，收到 475 条内容、写入 80 条新 RawItem，并发布了 36 个事件快照。

当前还存在两个次级边界问题：

1. `DailyReportService.generate(window_hours)`读取最新可见事件快照，而不是自动日报关联子任务的精确快照，存在并发运行串批风险。
2. 自动日报先归档并提交决策版音频，再校验和提交全览版音频。全览不就绪时会留下已归档空报告和单边音频。

首次运行出现的 SQLAlchemy 脱离会话错误已经由提交 `7345b4d` 修复，不属于本设计的待修范围。

## 方案比较

### 方案 A：复用高价值波次，推荐

自动日报冻结现有 `high-value-ai-tech` 波次计划，按页面选择的 24、48 或 72 小时覆盖计划窗口，然后排队 `high_value_news_wave`。该 Worker 已经具备并发抓取、来源级失败隔离、FetchRun/RawItem 持久化、精确批次选择范围和事件快照生成。

优点是复用已经真实验收的生产链路，改动集中、恢复语义明确，不新增抓取器。限制是第一版日报覆盖显式的 41 个高价值目标，而不是把 187 个目录目标都当成可抓取来源。

### 方案 B：自动日报逐个排队普通 Fetch Operation

为所有可抓来源创建独立 fetch 子任务，再等待全部任务结束并运行事件管线。

该方案会新增批次清单、子任务聚合、批次选择范围和恢复逻辑，而这些能力已经存在于高价值波次中，属于重复实现，因此不采用。

### 方案 C：让来源目录刷新同时写入 RawItem

把探测器升级为正式抓取器。

该方案会混淆控制面和数据面，使来源验收动作产生内容写入副作用，也会重复已有平台抓取器，因此不采用。

## 总体架构

来源目录刷新继续作为独立的来源可用性管理功能，不再作为每日内容采集入口。新自动日报链路如下：

```text
冻结窗口与高价值波次计划
→ 提交真实内容波次
→ 等待来源抓取和事件快照完成
→ 从该波次的精确快照生成日报草稿
→ 为决策条目和全览候选写入中文审核
→ 一次性校验并归档
→ 同一事务提交决策版和全览版音频
→ 等待两份音频并完成自动日报
```

第一版继续使用单个 Worker 可安全推进的短阶段续任务，不在父任务中阻塞等待子任务。

## 自动日报状态与兼容性

新增阶段：

- `enqueue_content_wave`
- `wait_content_wave`

新建自动日报直接从 `enqueue_content_wave` 开始。旧的 `enqueue_source_refresh`、`wait_source_refresh`、`enqueue_event_pipeline` 和 `wait_event_pipeline` 仍保留在枚举和运行器中，仅用于解释已有运行记录，不用于新运行。

高价值波次 Operation ID 存入现有 `DailyAutopilotRunRecord.event_operation_id`。该字段本来就是指向通用 OperationRun 的外键，而高价值波次本身已经包含事件处理结果，因此无需新增数据库字段或迁移。`source_operation_id`仅保留给历史运行。

父运行的 `requested_scope`保存无密钥的冻结 WavePlan：profile id、digest、window hours、trend days，以及每个成员的 source id、provider id、definition hash、roles、availability、access kind、fetchable 和 blocked reason。不得保存凭据值；只允许使用排队时已经计算出的可抓取结论。

## 波次计划与时间窗口

Web 边界加载 `wave_profiles/high-value-ai-tech.yaml`、当前来源定义、已持久化探测快照和成功抓取方法，并读取“已配置凭据名称集合”。页面选择的 24、48 或 72 小时覆盖 profile 的 `window_hours`后再调用现有 `build_wave_plan`，使 digest 和实际窗口一致。

自动日报运行过程中不得重新读取 `.env` 来改变成员是否可抓取；Worker只执行冻结计划。MiniMax不得参与成员选择。

如果已有高价值波次正在运行，自动日报保持在 `enqueue_content_wave`，延迟续排而不是创建重复波次。如果子波次最终为 `succeeded` 或 `partial`，并且 `event_manifest_complete`为真，则允许进入报告阶段。其他终态使用子任务已经持久化的中文诊断结束父运行。

## 内容采集和事件处理

继续使用 `HighValueWaveHandler`：

- 全局并发 6，同平台并发 2；
- 每个成员独立认领和终态记录；
- 使用现有 `execute_production_fetch`和平台抓取器；
- 网络请求受连接、读取和来源总超时约束；
- Worker检查取消和租约；
- 每条内容独立写入，单条格式或持久化错误不阻塞同一来源的后续内容；
- 结果保存 FetchRun ID、收到/新增/更新/未变化数量和中文失败原因。

波次事件阶段只选择本批成功或部分成功成员对应 FetchRun中出现的 RawItem，并按窗口过滤。现有事件管线继续负责相关性、新闻价值、实体、聚类、重复抑制、证据根、confirmed/emerging、排名和不可变事件版本。

MiniMax只辅助候选语义比较和事件中文增强。中文增强必须明确要求简体中文标题、中文概述和关注理由；模型超时、不可用或响应无效时继续使用规则回退，并在日报中标记 `minimax_degraded`。

## 精确日报快照

新增显式接口：

```python
DailyReportService.generate_from_operation(
    operation_id: int,
    window_hours: int,
    *,
    now: datetime | None = None,
) -> DailyReportRecord
```

该接口只接受状态为 `succeeded`或`partial`、事件清单完整、窗口一致的 `event_pipeline`或`high_value_news_wave`快照。自动日报必须调用此接口并传入自己的 `event_operation_id`。现有手动生成入口可以继续调用“读取最新完整快照”的 `generate(window_hours)`。

决策简报继续最多收录每栏 20 条，即 confirmed 最多 20 条、emerging 最多 20 条。情报全览保存该快照内所有 confirmed、hotspot和signal候选。页面每张卡片最多显示 3 个快捷原文链接，但完整报告证据区继续保留全部安全公开链接。

## 中文审核

自动审核沿用确定性规则，并保存为可追溯审核版本：

- confirmed或至少两个独立证据根：`keep`；
- 其他可展示事件：`needs_evidence`；
- 新闻价值不足在事件管线中排除；
- 已聚为同一事件的内容不重复生成日报条目；人工仍可将全览候选标记为`exclude`或`duplicate`。

自动审核使用事件快照中的中文标题、中文概述和关注理由，生成中文审核建议和证据评价。它不能改变来源合法性、事件证据根或确认状态。

## 空内容、部分成功和失败语义

必须区分以下情况：

1. 没有产生任何 FetchRun，或波次事件清单不完整：父运行`failed`，中文诊断指出采集链路未执行或未完成。
2. 至少有一个真实 FetchRun，但本窗口没有合格事件：父运行以`completed`阶段、`succeeded`状态结束，`result_summary.outcome = "no_content"`，不创建、归档日报或音频。
3. 部分来源失败但事件清单完整且有事件：继续生成日报，在页面展示成功、阻塞和失败来源数量。
4. 日报成功但某一音频失败：日报保持可用，父运行失败信息明确指向具体音频；另一份成功音频不得被删除。

状态表不新增`no_content`枚举，从而避免数据库迁移；无内容通过成功状态和结果摘要表达。

## 原子归档与双音频

新增成对命令边界，在任何写入前完成以下检查：

- 日报仍为草稿；
- 决策条目和全览条目都已审核；
- 至少存在一个可收录决策条目和一个可收录全览条目；
- 两份播报稿通过中文文本完整性检查。

全部检查通过后，在同一数据库事务中归档日报，并创建决策版和全览版两个`daily_report_audio` Operation。自动日报保存两个 Operation ID，再分别等待其终态。这样不会再出现空报告已归档、决策音频已生成、全览音频才发现不可用的半完成状态。

## 网页展示

自动日报详情页按实际数据链路显示：

- 时间窗口和高价值成员总数；
- 可抓取、成功、部分、阻塞、失败数量；
- FetchRun数量、收到、新增、更新和未变化条数；
- 事件候选、收录、排除、confirmed和emerging数量；
- 决策简报条目数和全览候选数；
- 决策版、全览版音频状态；
- 每个失败阶段的中文原因和下一步动作。

目录刷新187目标的统计继续出现在来源管理页面，不再显示为自动日报“已抓取新闻”的指标。

## 测试与验收

采用测试驱动实现，至少覆盖：

- 新自动日报冻结并排队高价值波次，不排队来源目录刷新；
- 24/48/72小时正确写入冻结 WavePlan和子波次；
- 子波次 partial但事件清单完整时继续；
- 自动日报使用自己的精确 Operation快照，不受更新快照干扰；
- 没有 FetchRun时失败，有 FetchRun但无事件时成功结束为`no_content`；
- 空报告不能归档，双音频只能成对排队；
- 单来源失败不阻塞其他来源和日报；
- 页面显示真实采集、事件、日报和音频指标。

真实验收必须新建一轮自动日报，并验证：FetchRun大于0；波次 RawItem收到数大于0或明确为no change；事件清单完整；有事件时日报来源 Operation等于该波次 ID；中文概述、审核建议、证据评价和原文链接可见；两份音频均可播放或各自显示可定位的中文失败原因。最后执行完整 pytest、ruff和浏览器验收。

## 非目标

- 不新增来源或抓取平台；
- 不绕过登录、验证码或反爬；
- 不实现定时日报或推送；
- 不让 MiniMax决定来源启用、合法性、证据根或事件确认；
- 不删除或改写已有失败运行 #1、#2和空报告 #7、#8，它们保留为审计历史。
