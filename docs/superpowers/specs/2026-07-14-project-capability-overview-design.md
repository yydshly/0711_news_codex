# News Codex 项目能力总览收口设计

## 1. 目标

将现有 `/sources` 收口为唯一的“项目能力总览”，让使用者不需要在十多个页面之间推断项目现状，就能直接回答：

- 目录登记了多少平台和具体目标？
- 多少目标完成了探测、具备试用资格或实际抓取成功？
- 当前已经沉淀多少 RawItem、事件和中文增强结果？
- MiniMax 是否配置、是否真实调用、是否正在降级？
- YAML 人工目录与 PostgreSQL 运行数据是否一致？
- 当前最重要的能力缺口和下一步是什么？

本次只做能力收口和可视化，不重做来源、抓取或事件架构。

## 2. 当前能力基线

设计基线取自 2026-07-14 的当前功能分支与本地 PostgreSQL：

| 指标 | 当前值 | 口径 |
| --- | ---: | --- |
| 有效 Provider | 68 | 67 个 Provider YAML，加上来源默认使用的 `independent` |
| YAML Target | 166 | 人工维护的具体账号、频道、仓库、订阅源或查询目标 |
| 直接覆盖 Target | 48 | `coverage_mode=direct` |
| 最新探测成功 | 60 | 每个 Target 最新一次内容探测结果为 `success` |
| 可试用抓取 | 37 | 通过现有严格 TrialDecision 的 Target |
| 实际成功抓取来源 | 28 | 至少有一次 `succeeded` 或 `no_change` FetchRun |
| RawItem | 364 | PostgreSQL 中的规范化原始条目 |
| 事件 | 18 | 2 个 confirmed，16 个 emerging |
| MiniMax 使用记录 | 0 | 当前数据库没有实际模型调用记录 |
| 实体 | 0 | 当前数据库没有实体提取结果 |

数据库当前有 68 个 Provider，与有效 Provider 集合一致。数据库有 168 个 Target，比 YAML 多出测试残留 `legacy-source` 和旧 ID `universe-youtube-1`。能力总览必须把这两个 Target 显示为“目录同步漂移”，不能把数据库残留冒充为当前来源目录。`independent` 是 Hugging Face Blog 和 Microsoft Research 当前使用的默认 Provider，不属于漂移。

## 3. 收口原则

1. **一个总览入口**：`/sources` 负责项目能力总览；其他页面继续作为下钻详情。
2. **目录、运行、产出分层**：不再使用一个“可用”数字概括全部状态。
3. **展示真实数据**：所有运行指标来自 PostgreSQL；人工目录规模来自 YAML 真相或与其等价的当前版本快照。
4. **暴露降级状态**：MiniMax 未配置、未调用、调用失败和规则回退必须明确区分。
5. **不扩大实现范围**：不新增表、不改迁移、不改 Worker、不改抓取器、不扩充 Target。
6. **中文优先**：页面名称、指标、解释、异常和下一步均使用中文；保留必要的 Provider、Target、RawItem 技术名词并附解释。

## 4. 页面结构

### 4.1 顶部结论

页面标题改为“项目能力总览”。第一屏显示三句话：

- 当前能感知什么：已经能够从 28 个实际抓取来源获取信息，并生成 RawItem 和事件线索。
- 当前不能保证什么：目录登记不等于可抓取，事件也不等于经过 MiniMax 中文增强或事实确认。
- 当前最重要缺口：真实抓取覆盖率、目录同步漂移、MiniMax 未运行和实体能力未产出。

### 4.2 能力流水线

使用一条从左到右、可换行的能力链展示：

```text
平台目录 → 具体目标 → 探测成功 → 可试用 → 实际抓取来源 → RawItem → 事件 → MiniMax 增强
```

每个节点同时展示：

- 数量；
- 一句中文口径；
- 指向现有下钻页面的链接；
- 若存在异常，显示警告状态而不是绿色成功状态。

### 4.3 四块能力卡片

页面按四个稳定层级组织，不再围绕临时修复批次组织：

1. **来源目录**：Provider、Target、直接覆盖、间接发现、受限目录。
2. **探测与抓取**：最新探测结果分布、可试用数量、实际成功抓取来源、FetchRun 结果分布。
3. **信息产出**：RawItem、来源数量、时间范围、confirmed/emerging 事件数量。
4. **AI 与系统能力**：MiniMax 配置状态、调用次数、最近结果、实体数量、最近活跃 Worker。

### 4.4 当前成果预览

总览页只展示少量真实成果，不复制详情页：

- 最近 5 条 RawItem：标题、来源、发布时间，链接到 `/items/{id}`；
- 最近 5 个事件：中文标题或当前回退标题、状态、热度，链接到 `/events/{id}`；
- RawItem 与事件为空时明确显示“尚未产出”，不得用示例数据填充。

### 4.5 主要缺口

按严重程度显示可执行问题：

- YAML 与数据库数量或 ID 不一致；
- 已登记 Target 与实际抓取来源差距；
- MiniMax 已配置但无使用记录，或完全未配置；
- 实体、事件或 RawItem 尚未产出；
- Worker 没有近期任务心跳；当前 Worker 只在租约任务期间写心跳，因此空闲时不能仅凭数据库判断进程已经停止；
- 最近操作失败或部分成功数量过高。

每个问题必须包含“这意味着什么”和“去哪里查看”，不在总览页直接执行写操作。

### 4.6 指标说明

页面底部固定解释以下术语：

- Provider：平台或服务，例如 GitHub、YouTube。
- Target：平台内的具体账号、频道、仓库、订阅源或查询目标。
- 探测成功：最近一次探测取得合格样本，不代表长期稳定。
- 可试用：满足当前无凭据、公开直连和字段规则。
- 实际抓取来源：至少完成一次真实 FetchRun。
- RawItem：尚未汇总成事件的规范化原始信息。
- 事件：由一个或多个 RawItem 聚合出的线索或确认事件。

## 5. 查询与数据边界

新增独立只读投影 `CapabilityOverviewView`，由查询层一次性组装，避免模板直接查询数据库。

投影包括：

- `catalog`：Provider YAML 数量、有效 Provider 数量、YAML/数据库 Target 数量及漂移 ID；
- `probe`：最新结果分布、试用资格和最近探测时间；
- `ingestion`：FetchRun 分布、实际抓取来源、RawItem 数量和时间范围；
- `events`：confirmed、emerging、事件总量和最近事件；
- `ai`：MiniMax 是否配置、ModelUsage/EventModelRun 数量、最近调用结果、实体数量；
- `runtime`：近期有任务心跳的 Worker 数量、OperationRun 状态分布；该指标表示最近执行活动，不冒充操作系统进程存活状态；
- `gaps`：由确定性规则生成的中文问题列表。

为了保持 YAML 为来源真相，Web 应通过一个可注入的目录快照读取器获得当前 Provider/Target ID。测试使用内存快照；生产读取现有 `providers/` 和 `sources/`，不修改文件。

所有查询必须是只读、有界查询。访问 `/sources` 不发起网络请求、不调用 MiniMax、不创建操作、不修改 YAML 或 PostgreSQL。

## 6. 导航与现有页面

- `/`：保留“事件情报”，展示用户最终消费的结果。
- `/sources`：改为“项目能力总览”，成为理解系统现状的唯一入口。
- `/providers`、`/targets`、`/probes`、`/operations`、`/fetch-runs`、`/items`、`/events`、`/system`：保持现有路由，作为总览下钻页面。
- `/remediation`：保留为历史修复审计页，不再承担项目总体状态说明。

侧栏只调整名称与分组，不删除已有路由，不改变任何写操作入口。

## 7. 错误与安全处理

- YAML 快照读取失败时，页面继续显示数据库运行数据，并标记“目录真相不可读取”。
- PostgreSQL 不可用时沿用现有服务不可用页面，不伪造零值。
- MiniMax 只显示是否配置和使用结果，不显示 API Key、请求正文或原始错误详情。
- URL、数据库连接串、代理、Cookie 和认证头不得进入页面。
- Worker 区域显示“近期有任务活动”而不是“进程存活”。不能只读取数据库中的 `status=running`，也不在 Web 进程中枚举操作系统进程。

## 8. 测试与验收

### 查询测试

- YAML 与数据库一致和不一致两种情况；
- 最新探测结果只按每个 Target 最新一次统计；
- 实际抓取来源按来源去重；
- `succeeded` 和 `no_change` 都计为完成抓取；
- RawItem、事件、实体、模型调用和 Worker 新鲜度统计准确；
- 无数据时返回明确零值和缺口，不抛异常；
- 敏感配置只返回布尔状态。

### 路由与模板测试

- `/sources` 返回 200，包含完整能力流水线；
- 页面同时展示目录、探测、抓取、RawItem、事件和 MiniMax 状态；
- 最近 RawItem 和事件可以下钻；
- YAML/数据库漂移和 MiniMax 未运行有中文提示；
- 页面不包含 API Key、DATABASE_URL、Cookie、Authorization 或代理值；
- 浏览 `/sources` 不产生数据库写入和网络请求。

### 浏览器验收

- 桌面和窄屏都能读懂流水线；
- 第一屏能够回答“现在能获得什么”；
- 所有数字都有口径说明；
- 从总览可进入 Target、抓取批次、RawItem 和事件详情；
- 不需要阅读 `/remediation` 才能理解项目整体能力。

## 9. 非目标

本次明确不做：

- 新增或重新审核来源；
- 修改 TrialDecision 或来源资格规则；
- 清理数据库残留数据；
- 调用 MiniMax 重新生成事件；
- 修改事件聚类、推荐、摘要或抓取调度；
- 新建另一套首页或新的能力路由；
- 重做前端技术栈或引入 Next.js。

## 10. 完成标准

完成后，使用者只打开 `/sources` 就能准确理解：

1. 项目登记了什么；
2. 实际探测和抓取到了什么；
3. 当前产生了哪些原始信息和事件；
4. MiniMax 与 Worker 是否真正运行；
5. 数据和能力还缺什么；
6. 每个问题应该进入哪个现有页面继续查看。
