# 日报归档触发标识修复设计

## 背景与问题

日报详情页提交“归档并生成决策版语音”时，Web 路由向持久操作队列传入 `daily_report_archive`。`operation_runs.trigger` 的数据库字段为 `VARCHAR(16)`，而该值长度为 20。PostgreSQL 会在插入语音操作时抛出字段超长错误；由于归档与入队位于同一事务，日报状态也会一起回滚为草稿。

SQLite 测试库不会强制执行 `String(16)` 长度，因此现有网页测试虽然覆盖了归档和入队流程，却没有发现生产 PostgreSQL 上的失败。

## 目标

- 让网页归档能够在 PostgreSQL 上原子地完成“归档日报 + 决策版语音入队”。
- 保持失败时整体回滚，不出现“已归档但没有语音任务”的半完成状态。
- 在进入数据库前拒绝过长的操作触发标识，并返回稳定、可测试的错误代码。
- 不修改日报内容、音频模型、队列结构或数据库字段。

## 不在本次范围内

- 不扩大 `operation_runs.trigger` 字段长度，不新增数据库迁移。
- 不静默截断触发标识，因为截断可能让不同触发来源变成相同值。
- 不改变已归档日报、已有音频制品或日报审核数据。
- 不调整全览版音频策略、MiniMax 参数或页面视觉布局。

## 方案比较

### 方案 A：仅缩短 Web 路由字符串

把 `daily_report_archive` 改为 `daily_archive`。改动最小，能够修复当前故障，但其他调用方将来仍可能传入过长值并直到数据库写入时才失败。

### 方案 B：缩短当前字符串，并在操作仓储入口校验（采用）

Web 路由使用 `daily_archive`；`OperationRepository.enqueue` 同时校验 `trigger` 是非空字符串且长度不超过 16。非法值在构造数据库记录前抛出 `ValueError("invalid_operation_trigger")`。

该方案既修复当前路径，也把数据库约束提升为应用层契约。它不需要迁移，且不会改变所有现有合法触发标识。

### 方案 C：扩大数据库字段

将字段扩大到更长的 `VARCHAR` 或 `TEXT`。这需要迁移，也无法防止触发标识无限增长或命名失控，因此不采用。

## 选定设计

### Web 路径

`POST /daily-reports/{report_id}/archive` 继续调用 `archive_and_enqueue_daily_report_audio`，但传入 `trigger="daily_archive"`。该值可读、稳定，长度为 13。

### 队列边界

`OperationRepository.enqueue` 在开启事务和添加 `OperationRunRecord` 之前验证触发标识：

- 必须是字符串；
- 去除首尾空白后不能是空字符串；
- 原值长度不能超过 16；
- 失败统一抛出 `ValueError("invalid_operation_trigger")`；
- 不自动清洗、截断或替换调用方传入的值。

合法值保持原样写入，避免无关行为变化。数据库字段仍作为最终防线。

### 事务与错误处理

日报归档和自动创建决策版语音操作继续使用现有单事务实现。若操作仓储校验、数据库写入或其他入队步骤失败，归档状态随事务一起回滚。Web 层沿用现有日报错误处理，不新增网络请求或重试。

## 测试设计

按测试驱动顺序先增加失败测试，再实现修复：

1. 网页归档测试断言响应为 303，日报变为 `archived`，并且生成的语音操作 `trigger == "daily_archive"`、长度不超过 16。
2. 操作仓储测试断言长度超过 16、空字符串和非字符串触发值均抛出 `invalid_operation_trigger`，且不会写入操作记录。
3. 保留并运行现有事务回滚测试，证明自动语音入队失败时日报仍为草稿。
4. 保留现有重复语音任务复用测试，证明本修复不改变幂等行为。
5. 完成后运行相关测试、完整 `pytest` 与 `ruff check .`。

## 验收标准

- PostgreSQL 上通过网页归档日报时不再发生 `VARCHAR(16)` 超长错误。
- 归档后恰好存在一个决策版语音操作，其触发标识为 `daily_archive`。
- 非法触发标识在数据库写入前得到稳定诊断。
- 所有既有测试与静态检查通过。
- 不读取或修改 `.env`，不触碰用户保留报告，不合并或推送分支。
