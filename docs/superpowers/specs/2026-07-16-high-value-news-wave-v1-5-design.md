# News Codex v1.5 高价值真实热点波次设计

## 1. 背景与目标

v1.4 已完成 67 个 Provider、187 个具体来源目标的目录分层、真实探测、冻结批次、Worker 可靠性和中文全量盘点。当前计划中有 106 个内容通道目标、20 个能力通道目标和 61 个目录通道目标。

v1.5 不再扩张来源目录或重写事件架构，而是从现有高价值目标中选择约 30–50 个入口，打通真实新闻产品闭环：

```text
高价值来源 → RawItem → 规范化与去重 → 发现信号 → 事件聚类
→ 事实证据确认 → 综合热度排序 → 24 小时热点与 7 天趋势
```

用户最终获得的不是“来源是否能访问”的技术报告，而是可追溯、可解释的中文 AI/技术热点：现在发生了什么、为什么热门、哪些来源在讨论、事实由谁确认、信息是否仍有分歧。

## 2. 已确定的产品原则

- 默认展示最近 24 小时热点，并附 7 天趋势。
- 热点按综合热度排序，不只按时间、点赞或来源权威性排序。
- 社区和社交平台可产生“早期信号”，但不能独立确认新闻事实。
- 早期信号与已确认热点分区展示；确认后才升级状态。
- 规则、来源证据和持久化数据决定基础排名与确认状态。
- MiniMax 只辅助中文表达、主题分类和受约束的事件合并判断；模型不可用时系统仍可运行。
- 复用现有 RawItem、Event、EventVersion、事件快照、Worker、操作日志和网页，不创建第二套新闻系统。

## 3. 首轮来源范围与角色

首轮只选择现有目录中已经登记的高价值目标，不以增加目标数量作为验收指标。

### 3.1 发现与讨论热度

- Hacker News：topstories、newstories、beststories。
- Reddit：LocalLLaMA、MachineLearning、artificial；无凭据时保持能力阻塞，不回退网页抓取。
- YouTube：已确认的官方公司、研究机构和技术频道。
- Bluesky、Mastodon：已确认的公司、研究员和开发者账号。

这些来源提供早期发现、传播速度、互动量和开发者反馈，默认属于 `discovery` 或 `engagement`，不能单独确认事实。

### 3.2 聚合发现

- Techmeme。
- Google News RSS。
- GDELT。

聚合源用于发现“多个媒体正在集中报道”，必须解析或关联原始发布者；聚合条目本身不计作独立事实证据。GDELT 的 429、超时和实体歧义必须如实保留。

### 3.3 专业媒体证据

- Reuters、AP、BBC、The Guardian。
- TechCrunch、The Verge、Wired、Ars Technica、MIT Technology Review、VentureBeat 等现有可用媒体入口。

专业媒体主要用于独立证据和背景。转载、联合稿、同集团重复内容只计算一个独立根来源。

### 3.4 官方与开发来源

- 模型厂商官方博客、公告和更新日志。
- GitHub Releases、仓库活动。
- arXiv、OpenReview、研究机构页面。

官方来源可以确认自身产品发布、版本、政策和研究材料，但“厂商自述”不自动等于行业影响评价。

## 4. 事件状态与确认规则

事件对外状态固定为：

- `early_signal`：社区、社交或聚合源正在升温，但事实证据不足。
- `confirmed`：具有符合规则的官方或独立专业媒体证据。
- `disputed`：可靠来源之间存在尚未解决的事实冲突。
- `cooling`：24 小时热度已经下降，但仍保留在 7 天趋势。

确认规则：

1. 官方一手来源可以确认自身产品发布、版本和公告。
2. 非官方事实需要两家独立专业媒体，或一家独立专业媒体加一份可核验的一手材料。
3. 社交、社区、聚合源不能单独把事件升级为 `confirmed`。
4. 同一原稿的转载、聚合跳转和联合稿只计算一个证据根。
5. 证据撤回、链接失效或来源归属改变时，事件需要重新评估；不得保留过期确认结论。

现有事件 `display_tier`、证据质量和版本快照继续作为持久化基础。v1.5 在读取层建立面向用户的状态映射，只有无法表达所需事实时才新增数据库字段。

## 5. 数据流与组件边界

### 5.1 高价值来源选择器

新增一个持久化、可审计的“高价值波次配置”，只引用现有 `source_id`，不复制来源 URL 或凭据。选择器根据来源状态、角色、最新探测、访问方式和人工纳入理由生成冻结清单。

首轮配置必须满足：

- 七类来源角色都有代表入口。
- 受限来源可以出现在能力说明中，但不能进入内容抓取成员。
- 当前访问方式与最新探测方式不一致时禁止入队。
- 来源定义变化后旧操作保持冻结语义。

### 5.2 正式抓取与 RawItem

复用现有 fetch operation、协议 fetcher、`RawItemRepository` 和幂等快照。每个高价值波次只对冻结成员发起抓取；单来源失败不阻塞其他来源。

RawItem 必须保存或可追溯：来源 ID、Provider、原始 URL、Canonical URL、发布时间、抓取时间、作者、标题、摘要或正文片段、互动指标、来源角色以及 fetch operation。

去重优先级：

1. Canonical URL 与最终原始发布者。
2. 来源内稳定 ID、GUID、视频 ID、帖子 ID。
3. 规范化标题与时间窗口指纹。
4. 聚合跳转解析后的原始 URL。

不得因文本相似就删除来源独立性；去重解决重复条目，事件聚类解决多来源报道同一事件。

### 5.3 候选信号与事件聚类

复用现有 EventPipeline、实体、相关性、聚类、pairing 和不可变事件版本。v1.5 重点补齐来源角色与互动特征，使社区信号、专业证据和官方证据在同一事件中承担不同作用。

规则聚类先使用：规范化实体、产品/项目名、链接根、时间窗口、关键动作词和标题相似度。MiniMax 只在边界候选上提供结构化 `same_event` 判断；低置信度、非法响应或模型失败回退规则结果。

### 5.4 证据解析器

证据解析器从事件成员生成去重后的证据根：

- `first_party`：官方一手。
- `independent_media`：独立专业媒体。
- `community_signal`：社区或社交信号。
- `aggregator_pointer`：聚合指针。

它输出确认状态、确认理由、缺失条件、分歧说明和证据 URL，不生成不可追溯的模型结论。

### 5.5 综合热度与趋势

综合热度采用确定性快照，输入包括：

- 传播速度：短时间新增报道或讨论速度。
- 独立来源覆盖：不同 Provider、不同证据根数量。
- 社区互动：可获得的点赞、评论、转发或投票指标。
- 新鲜度：首次发生、最近升温和快照时间。
- 证据质量：官方或独立媒体确认强度。
- 惩罚项：转载重复、单社区集中、聚合未还原、字段缺失和异常互动。

排名输出必须包含分项分数与中文理由。早期信号单独排序，不能依靠高互动挤入已确认热点榜。

7 天趋势根据连续不可变快照判断：`rising`、`sustained`、`cooling`。不得直接使用当前墙上时间重算历史快照。

## 6. 网页与用户操作

主要增强现有 `/events` 和事件详情，不新建平行热点产品。

### 6.1 事件首页

页面顶部显示：

- 最近一次成功快照时间。
- 本轮覆盖来源、成功、失败、阻塞数量。
- 已确认热点数量、早期信号数量、存在分歧数量。
- “更新高价值来源”操作入口及当前 Worker 状态。

主体分为：

- 最近 24 小时已确认热点。
- 最近 24 小时早期信号。
- 7 天趋势。

事件卡片显示中文标题、状态、热度分、趋势、独立来源数、社区信号数、确认来源数和主要主题。

### 6.2 事件详情

详情展示：

- 中文解释与原始主题。
- 事件时间线。
- 为什么热门及分数构成。
- 发现来源、互动来源、确认来源和聚合指针。
- 尚未满足的确认条件。
- 来源分歧和已知限制。
- 所有可审计的原始链接。

### 6.3 操作方式

第一轮提供网页“一键更新高价值来源”和 CLI 入队，复用现有 Operation/Worker。连续三轮真实验收通过后，再启用本机定时运行：

- 高时效社区与社交来源：15–30 分钟。
- 专业媒体、聚合和官方来源：30–60 分钟。

本阶段不绑定 Windows Task Scheduler 的具体部署；先提供幂等的 `enqueue-due` 能力和可观测状态，部署方式在验收后决定。

## 7. MiniMax 使用边界

使用 `MiniMax-M2.7-highspeed`：

- 批量中文标题翻译。
- 主题分类与标签补全。
- 规则边界上的事件配对判断。

使用 `MiniMax-M3`：

- 已确认事件的中文解释、影响和来源分歧整理。

模型约束：

- 输入内容始终标记为不可信数据。
- 模型无工具、文件、网络和环境变量访问权限。
- 所有响应由 Pydantic 校验；结构错误最多修复一次。
- 调用失败回退规则结果。
- 模型不决定来源合规、事实确认或基础热度分。
- 记录目的、模型、token、耗时和错误，不记录 Key。

## 8. 可靠性、错误与日志

- 单来源失败不影响其他来源、RawItem 持久化和事件快照。
- 429 尊重 Retry-After 与操作 deadline；权限、审批、付费不重试内容请求。
- 复用 claim fencing，旧 Worker 不能覆盖新 attempt；安全优先恢复必须显式确认。
- RawItem、事件聚类和发布均幂等，可重复执行而不重复创建。
- 日志关联 operation_id、source_id、raw_item_id、candidate_id、event_id 和 model_run_id。
- 网页、CLI、报告和日志使用一致的中文错误分类。
- Authorization、Cookie、API Key、数据库连接和完整正文不进入日志或诊断页面。

## 9. 测试与真实验收

### 9.1 自动化测试

- 高价值选择器：角色覆盖、冻结、定义漂移、受限来源排除。
- Fetch/RawItem：协议固定样本、幂等、Canonical URL、聚合原始 URL。
- 事件：社区早期信号、官方确认、双媒体确认、转载根去重、分歧状态。
- 排名：综合分组成、早期信号隔离、7 天趋势快照稳定。
- MiniMax：非法 JSON、超时、429、模型完全离线回退。
- Worker：并发、取消、deadline、claim fencing、安全恢复和单来源失败继续。
- Web：中文状态、筛选、分数解释、无快照降级、写操作安全边界。

### 9.2 真实验收

1. 使用现有高价值目标连续运行三轮。
2. 每轮记录冻结来源、RawItem 新增/重复、事件新增/更新、失败来源和耗时。
3. 人工抽查至少 20 个事件或当轮全部事件（取较小者）。
4. 确认聚合稿不冒充独立证据、社交热点不冒充已确认事实。
5. 确认相同新闻不会形成多个重复事件。
6. MiniMax 完全关闭后重新运行一轮，热点页仍可用。
7. 生成中文验收报告和浏览器证据。

完成条件：

- 24 小时已确认热点、早期信号和 7 天趋势均能展示真实数据。
- 已确认事件均可追溯到合规证据；早期信号明确说明缺少什么。
- 连续三轮无批次卡死、无重复 RawItem/事件爆炸、无模型依赖阻断。
- 页面、Worker、日志与报告对同一失败给出一致解释。

## 10. 非目标

- 本轮不新增大量账号、频道或媒体栏目。
- 不接入 Cookie、登录态爬虫、代理绕过或验证码破解。
- 不实现个性化推荐、飞书/邮件推送或移动端。
- 不以 MiniMax 摘要数量作为成功指标。
- 不重写现有 RawItem/EventPipeline，不建立第二套事件表。
- 不直接启用无人值守定时任务；需先通过三轮人工触发验收。

## 11. 实施边界

实施按垂直闭环拆分：高价值选择与入队、来源角色特征、证据状态、热度趋势、中文页面、真实验收。每个阶段必须复用现有组件并通过回归测试后再进入下一阶段。

设计与实施使用隔离分支 `codex/high-value-news-wave-v1-5`。设计审查推荐使用 5.6 Sol + 高推理；实现任务推荐使用 5.6 Terra + 中推理；最终全分支审查使用 5.6 Sol + 高推理。
