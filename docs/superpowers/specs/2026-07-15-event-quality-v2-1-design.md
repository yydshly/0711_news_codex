# News Codex 事件质量 v2.1 收口设计

日期：2026-07-15

状态：设计已确认，待实施计划

适用分支：`codex/event-quality-v2-1`

## 1. 背景与真实基线

Event Intelligence v2 已经完成并合并。本设计不重建事件系统，而是在现有
`RawItem → EventCandidate → Event → EventVersion` 链路上解决真实数据暴露出的质量问题。

2026-07-15 本地 PostgreSQL 基线：

- 273 个事件，其中 199 个 `current`、74 个 `legacy`；
- 199 个当前事件中有 195 个 `emerging`、4 个 `confirmed`；
- 185 个当前事件只有一条成员信息，13 个有两条，1 个有三条及以上；
- 198 个当前事件只有一个独立证据根；
- 最近 72 小时 411 条 RawItem 已实现 100% `relevance-v2` 处理覆盖；
- 411 条中 206 条纳入、205 条排除；
- 当前事件只有 1 个完成 MiniMax 中文增强，其余 198 个为规则回退；
- 当前列表仍包含泛科技、弱事件性内容、自动转发文本和未被聚合的同一新闻报道。

这说明来源获取和事件处理已经能够运行，但用户面对的仍是“很多单条线索”，而不是少量、清晰、
可确认的 AI/技术热点。

## 2. 已确认的产品决策

1. 采用“热点事件、新兴线索、噪声审计”三层结构。
2. 官方一手重大事件允许凭一个独立官方证据根成为已确认热点。
3. 专业媒体消息需要至少两个独立专业媒体证据根才能成为已确认热点。
4. 社交、社区和聚合来源只提供发现、讨论和热度信号，不能单独确认事实。
5. 研究预印本默认属于研究线索，并明确标注“未经同行评审”。
6. MiniMax 自动增强热点和高价值线索，其余内容按需增强。
7. 首页采用“综合热点 + 产品与模型 + 研究进展 + 开发者工具 + 公司与产业 + 新兴线索入口”。
8. 采用规则漏斗与有限模型辅助的混合聚类，不采用模型自由聚类。

## 3. 目标与非目标

### 3.1 目标

- 把同一事实的官网公告、媒体报道、聚合入口和社交讨论合并为一个可演进事件。
- 把用户主要阅读面从近 200 个单条事件收口为约 10–20 个可信热点和独立的新兴线索流。
- 为每个合并、分层、确认和排序结果提供稳定原因代码和中文解释。
- 保持模型可选：MiniMax 完全不可用时，规则管线仍可发布、查询和审计。
- 复用现有 Operation、Worker、心跳、租约、重试、取消和恢复能力。
- 保留全部 RawItem、旧事件、事件版本、处理结论和人工操作记录。

### 3.2 非目标

- 不新增来源平台或 Target。
- 不重新实现 RawItem 抓取、来源资格或 Provider 探测。
- 不实现个性化推荐、日报、推送、邮件或后台调度器。
- 不让模型决定事实真假、来源权利、证据独立性或最终确认状态。
- 不为了降低单条事件比例而强制合并不相关信息。
- 不删除或覆盖 Event Intelligence v2 的历史数据。

## 4. 总体架构

```text
RawItem
  ↓ relevance/newsworthiness v2.1
有效信息 ───────────────→ 噪声审计
  ↓ 确定性候选生成
强匹配 / 边界候选 / 明确不匹配
  ↓              ↓
直接合并      MiniMax 边界辅助
  └───────┬──────┘
          ↓
证据根归一化与事件版本发布
          ↓
热点事件 / 新兴线索 / 审计态
          ↓
重点中文增强与分类排序
```

v2.1 继续由 `EventPipeline` 编排。新增逻辑以独立纯函数和小型服务实现，不在网页请求中运行网络或模型调用。

## 5. 数据模型

### 5.1 事件展示层级

新增固定枚举 `EventTier`：

- `hotspot`：满足事实确认和首页质量门槛的热点；
- `signal`：证据不足但值得追踪的新兴线索；
- `audit_only`：被排除、人工拒绝或仅保留审计的事件结果。

`events` 增加 `display_tier`，表示当前完整版本的物化层级并供索引查询；同一层级同时写入
`event_versions.payload.publication.tier`，确保历史版本可以重放和审计。`status` 继续表示
`confirmed/emerging/developing/disputed/stale/rejected`，`visibility` 继续表示 `current/legacy`，三者不能混用。

迁移规则：现有 `current + confirmed` 回填为 `hotspot`，现有 `current + emerging/developing/disputed`
回填为 `signal`，`rejected` 和 `legacy` 回填为 `audit_only`。回填不改变原状态、版本和成员关系。

### 5.2 候选对判定审计

新增 `event_pair_decisions`：

- 无序的 `left_raw_item_id`、`right_raw_item_id`；
- `algorithm_version`、`input_fingerprint`；
- 规则得分与稳定原因代码；
- 可选的 `model_usage_id`、模型结论与置信度；
- 最终结论 `merge/separate/undetermined`；
- 创建时间。

唯一键为无序 RawItem 对、算法版本和输入指纹。相同输入不得重复调用模型；输入发生实质变化时产生新记录，
不覆盖旧记录。输入指纹只对规范化标题、受限摘要、核心对象、动作、时间桶和证据根标识计算哈希，
不得包含完整 payload、凭据或 URL 查询参数。

## 6. 相关性与新闻价值

v2.1 在现有 AI 相关性之后增加 `newsworthiness-v1`。一条信息必须同时具备 AI/技术相关性和可识别事件动作，
才进入聚类候选。

事件动作至少覆盖：发布、更新、开源、融资、收购、合作、政策、研究结果、安全事件、价格变化、服务中断、
基准结果和重要人物公开主张。

以下内容进入噪声审计：

- 只有 AI 标签、链接或自动转发模板，没有事实主张；
- 泛科技、娱乐、游戏或普通商业内容仅偶然包含歧义词；
- 广告、订阅引导、招聘和无实质内容的宣传；
- 文本不足以识别主体、动作或对象；
- 已知非目标主题或与 AI/技术热点无直接关系的内容。

所有结论保存 `outcome`、0–100 分数、稳定原因代码和安全详情摘要。

## 7. 混合聚类

### 7.1 候选生成

只比较时间窗口内且至少共享一个阻塞键的条目：

- 相同 Canonical URL、原始报道 URL、论文 ID、仓库/Release ID、视频 ID；
- 相同规范化核心对象；
- 相同高置信标题指纹；
- 聚合或社交入口解析到同一原始证据根。

普通事件使用 72 小时窗口；持续发展事件允许在相同核心对象和动作族前提下扩展到 7 天。不得全量两两比较。

### 7.2 最终判定

- 精确标识或相同原始证据根：直接合并；
- 规则分数不低于 0.80，且存在结构锚点：直接合并；
- 规则分数不高于 0.45：直接分开；
- 0.46–0.79：进入 MiniMax 边界辅助；
- 缺少结构锚点时，即使模型判断相同也不得合并。

结构锚点指相同核心对象、相同事件动作族、明确原始证据关系中的至少一项。模型辅助结果只有在
`same_event=true`、置信度不低于 0.85，并与规则结构锚点不冲突时才能使最终结论成为 `merge`；
超时、限流、结构错误或无法判断时保持 `separate/undetermined`。

### 7.3 证据根

官网原文、媒体原文、论文、Release 或视频是根证据。Google News、Techmeme、GDELT、社交转发和社区讨论
在能够解析原始 URL 时继承该根；无法解析时只能作为发现或互动信号。一个根证据无论被多少聚合器转载，
只计一次独立覆盖。

## 8. 发布层级与确认规则

### 8.1 热点事件

热点必须满足 `visibility=current`、完整事件版本、AI 相关性不低于 70、具有明确事件动作，并满足以下之一：

- 至少一个独立官方一手证据根；
- 至少两个互相独立的专业媒体证据根。

单一专业媒体独家、社交/社区/聚合信号、预印本和证据冲突事件不能直接进入热点。

### 8.2 新兴线索

下列内容进入 `signal`：

- 单一专业媒体报道或独家；
- 只有社交、社区或聚合发现信号的高相关内容；
- arXiv、OpenReview 等未经额外确认的研究；
- 事实可能重要但证据数量、独立性或一致性尚不足的事件。

新证据到达后，原事件发布新版本并重新计算层级，不创建重复事件。

### 8.3 噪声审计

噪声主要保存在 RawItem 处理结论中。已经形成事件但被人工排除、质量规则拒绝或确认无关时，事件进入
`audit_only`。审计层不出现在首页或普通事件流，但可从质量报告和历史入口查看。

## 9. 排序与栏目

热点排名分数固定为：

```text
rank_score =
  0.30 × credibility
+ 0.25 × importance
+ 0.20 × recency
+ 0.15 × source_coverage
+ 0.10 × engagement_velocity
```

所有输入来自确定性评分快照。模型不能修改输入或权重。单一事件只占一个位置；同一原始证据的转载不得增加覆盖。

首页最多展示 20 个综合热点，不足时显示真实数量。四个分类栏目分别展示产品与模型、研究进展、开发者工具、
公司与产业；当某分类存在合格事件时，综合热点最多为该分类保留两个位置，但不使用低质量事件填充名额。

新兴线索使用独立榜单，优先考虑 AI 相关性、时间、可信发布者和真实互动增长，不与已确认热点直接竞争。

## 10. MiniMax 调用边界

自动调用：

- 准备进入 `hotspot` 的事件中文标题、摘要和关注理由；
- `rank_score >= 60` 的高价值新兴线索中文解释；
- 聚类规则边界区间内的候选对语义辅助。

其他事件只在用户点击“补充摘要”时调用。自动增强使用 `MiniMax-M2.7-highspeed`；深度冲突解释保留给
`MiniMax-M3`。每个候选最多发送 5 条受限证据摘要，不发送完整 payload、凭据、环境变量或带敏感查询参数的 URL。

每次调用记录目的、模型、token、延迟、结果和安全错误码。相同目的、模型、提示版本和输入指纹复用已有成功结果。

## 11. 可靠性与错误处理

- 网页只创建 Operation，Worker 执行聚类和模型任务；
- 模型并发上限为 2，单请求保持现有有界超时；
- 401/403、429、5xx、超时、截断和结构错误分别记录稳定错误码；
- 单个候选失败不影响同批其他候选；
- 中文增强失败时发布 `rule_fallback`；
- 聚类辅助失败时保守地不合并；
- 数据库写入使用短事务，网络和模型调用期间不持有事务或事件租约；
- Operation 保留检查点、心跳、取消、重试和过期租约恢复；
- 重放固定 `window_end`、算法版本和输入快照，保证确定性部分结果一致。

## 12. 网页体验

### 12.1 首页

首页显示：

- 今日综合热点；
- 四个分类栏目；
- 新兴线索数量与入口；
- 最近处理时间、热点数、线索数和噪声排除数；
- “已确认”“证据不足”“未经同行评审”“MiniMax 增强”“规则回退”等中文标签。

### 12.2 事件详情

详情页按角色展示官方证据、专业媒体、研究、社区讨论、社交热度和聚合发现，解释：

- 哪些条目被合并以及合并依据；
- 为什么确认或为什么仍是线索；
- 独立证据根、当前评分和排名依据；
- 事件发展时间线和新版本变化；
- MiniMax 调用状态和易读耗时，例如 `22.5 秒`。

继续提供“补充摘要、重新聚类、排除事件”受控操作，所有写操作进入 Worker。

### 12.3 质量观察

质量页增加：单成员/多成员事件、独立证据根分布、直接规则合并、模型辅助合并、候选分开、模型降级、
热点/线索/审计数量和主要排除原因。能力探测成功不能冒充事件内容覆盖。

## 13. 测试与验收

### 13.1 自动测试

- 新闻价值正反样本与稳定原因代码；
- 精确标识、原始证据、实体动作、标题和时间窗口聚类；
- 规则高低阈值和模型边界区间；
- 模型高置信但无结构锚点时不得合并；
- 聚合转载证据根去重；
- 官方单根、双媒体、社交单源和预印本的分层规则；
- 排名公式、分类保留位和不足时不填充；
- 模型缓存、并发上限、超时、429、5xx、非法结构和降级；
- Operation 重试、取消、心跳和租约恢复；
- current/legacy、hotspot/signal/audit_only 查询和中文页面；
- 迁移无损保留全部旧事件、版本、成员、评分和模型记录；
- 凭据、Cookie、连接串、提示全文和敏感 URL 不进入数据库审计、日志、报告或 HTML。

### 13.2 真实数据验收

- 最近 72 小时 RawItem 处理覆盖率为 100%；
- 建立至少 50 个同事件正例对和 50 个不同事件反例对的人工标注回归集；正例合并召回率不低于 85%，
  反例不得误合并；
- 首页前 20 条人工抽检准确率不低于 90%；
- 同一新闻的官网、媒体、聚合和社交样本能够形成一个事件及多个证据角色；
- current 单成员比例作为观察指标下降，但不以强制合并方式追求固定数值；
- 至少一个热点和一个高价值线索完成真实 MiniMax 中文增强；
- MiniMax 完全不可用时，同一批次仍能完成并生成规则回退结果；
- 网页到 Worker 的补充摘要、重新聚类、取消和失败定位链路通过验收；
- 生成中文 v2.1 质量报告，记录输入、排除、候选对、合并、分层、排名、模型和遗留问题。

## 14. 实施边界

实施必须在独立分支和工作树完成。不得修改或删除 `main` 中用户未提交的来源报告，不得删除包含 `.env`、
`.local/postgres` 或本地运行数据的既有工作树，不得强制推送。设计落地后再编写中文实施计划，并按测试驱动方式分阶段实现。
