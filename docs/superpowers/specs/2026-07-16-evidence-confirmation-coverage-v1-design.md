# News Codex「证据确认覆盖 v1」设计规格

## 1. 文档目的

本阶段承接 v1.5「高价值真实热点波次」。v1.5 已打通来源抓取、RawItem、事件聚类、证据判断、中文事件页和运行观测，但真实验收暴露了一个明确缺口：最近一次真实操作覆盖 35 个目标，24 个成功、11 个阻塞，生成 32 个早期事件，却没有形成 `confirmed` 事件。

问题不在于页面缺失，也不在于抓取框架尚未建立，而在于当前波次中可直接参与事实确认的官方和专业媒体入口过少；大量候选来自 Google News 等聚合发现源，能够说明“有人在报道”，却不能被误算为独立证据。

因此，本阶段只完成一件事：在不降低证据标准、不重建系统的前提下，提高同一真实事件被官方或多家独立专业媒体命中的概率，并使跨来源报道能够安全聚合到同一事件。

阶段名称固定为：**证据确认覆盖 v1**。

## 2. 当前基线与问题判断

### 2.1 已有能力

- 来源目录已经区分 Provider 与具体 Target，并记录来源性质、角色、访问方式、风险和探测结果。
- Worker 已能并发消费 fetch 任务，并支持超时、取消、重试、心跳、恢复和单来源失败隔离。
- 抓取结果统一进入 `RawItem`，具备 Canonical URL、原始 URL、来源和操作批次等审计字段。
- `EventPipeline` 已具备候选生成、规则聚类、事件版本、证据根判断和中文页面展示能力。
- 当前证据规则已经正确限制：聚合、社区和社交来源不能独立确认新闻事实。
- MiniMax 不可用时，抓取、规则聚类、证据判断和页面仍可运行。

### 2.2 真实验收结论

最近一次 v1.5 真实操作为 `#963`：

- 冻结目标：35 个。
- 成功：24 个。
- 阻塞：11 个。
- 早期事件：32 个。
- 已确认事件：0 个。

当前事件多数由 Google News 聚合条目产生，事件证据摘要中常见 `aggregator=1/2`、`professional=0`。这说明发现层正在工作，但确认层的直接命中不足。系统不应通过放宽规则把聚合条目包装成专业媒体证据，而应补充真正可审计的官方/专业媒体入口并提高同事件跨来源匹配召回率。

## 3. 目标与成功标准

### 3.1 目标

1. 将 v1.5 高价值波次从 35 个目标扩展到 41 个目标。
2. 新增 6 个已完成初步接入审核、可直接访问、探测成功且具备 `evidence` 角色的来源。
3. 改进现有事件聚类，使同一事件的官方报道和专业媒体报道能够安全归并。
4. 保持聚合、社交、社区来源只承担发现或热度角色。
5. 连续三轮真实抓取中，至少有一轮自然产生一个可重复、可追溯的 `confirmed` 事件。
6. 网页、CLI、运行报告和事件版本对确认原因给出一致的中文解释。

### 3.2 数据验收底线

一个事件只有在满足下列任一条件时才可标记为 `confirmed`：

- 存在与该事件直接相关的官方一手证据根；或
- 存在两个相互独立的专业媒体证据根。

若三轮内没有自然出现满足条件的事件：

- 不伪造样本；
- 不降低证据阈值；
- 不把聚合转载算作独立证据；
- 不把 MiniMax 判断当作事实确认；
- 阶段结论记为“工程验收通过、数据验收未通过”，继续定位来源命中率和聚类召回率。

## 4. 范围与非目标

### 4.1 本阶段范围

- 调整现有高价值 Profile 的冻结成员。
- 复用现有 RSS/API/官方公开访问方式抓取新增来源。
- 增强现有候选配对、跨来源聚类和证据根去重。
- 在现有首页、事件详情、运行页和中文报告中补充证据覆盖指标。
- 完成自动化测试、三轮真实抓取和人工抽查。

### 4.2 明确不做

- 不创建第二套抓取系统、RawItem 表或 Event 表。
- 不重做导航、首页信息架构或事件详情页。
- 不新增自动 HTML 抓取、登录页抓取或付费墙抓取。
- 不使用 Cookie、登录态、验证码破解、代理绕过或非官方高风险爬虫。
- 不新增定时调度、摘要推荐、消息推送或个性化功能。
- 不把五个 arXiv 分类加入本轮产品/商业新闻确认波次；它们继续留在研究信息体系。
- 不让 MiniMax 决定来源角色、证据独立性或事件确认状态。

## 5. 总体架构

本阶段复用现有垂直闭环：

```text
扩展后的高价值 Profile（41 个目标）
  -> 冻结来源定义与访问方式
  -> Worker 并发抓取
  -> RawItem 规范化与幂等保存
  -> 同一事件跨来源聚类
  -> 独立证据根判断
  -> confirmed / emerging
  -> 现有中文首页、事件详情、运行页和报告
```

不引入旁路数据模型。任何新增判断结果必须进入现有不可变事件版本或现有运行快照，使历史结果能够按当时的来源定义、算法版本和证据状态复现。

## 6. 来源覆盖设计

### 6.1 Profile 扩展

当前高价值 Profile 从 35 个唯一 Target 扩展为 41 个。新增来源固定为：

| 来源 ID | 类型 | 主要作用 | 获取方式要求 |
|---|---|---|---|
| `google-ai-blog` | 官方一手 | Google AI 产品与研究发布确认 | 已审核的直接 RSS/API 方式 |
| `nvidia-developer-blog` | 官方一手 | NVIDIA 开发者产品、平台和技术发布确认 | 已审核的直接 RSS/API 方式 |
| `universe-cnbc-1` | 专业媒体 | 商业、公司与市场事件证据 | 已审核的直接 RSS/API 方式 |
| `universe-mit-tech-review-1` | 专业媒体 | AI 技术与影响报道证据 | 已审核的直接 RSS/API 方式 |
| `universe-venturebeat-1` | 专业媒体 | 企业 AI 与开发工具报道证据 | 已审核的直接 RSS/API 方式 |
| `universe-wired-1` | 专业媒体 | 科技产品、政策与社会影响报道证据 | 已审核的直接 RSS/API 方式 |

这里的“初步接入审核”指身份、公开入口、抓取许可边界和固定样本已经满足本轮使用条件，不代表来源深度审计已经完成；YAML 中的 `research.status: needs_research` 可以保留，并在后续来源深度迭代中继续补齐。

这些来源必须在进入冻结 Profile 前同时满足：

- YAML 定义已同步且没有定义漂移；
- 当前访问方式为直接内容方式，而非目录占位或间接发现；
- 最新探测成功；
- 来源角色包含 `evidence`；
- 没有凭据、审批、付费或人工访问阻塞；
- 不依赖 HTML 回退。

### 6.2 现有证据来源继续保留

当前波次中已存在的证据型来源继续参与，包括但不限于：

- 官方：`openai-news`；
- 专业媒体：`universe-ars-technica-1`、`universe-bbc-1`、`universe-guardian-1`、`universe-techcrunch-1`、`universe-the-verge-1`。

最终 41 个成员以冻结快照为准，运行中的目录变化不能改变历史批次语义。

### 6.3 来源角色约束

- 官方一手来源可以确认自身产品发布、版本、政策、研究材料和公告。
- 专业媒体可以形成独立媒体证据根，但转载、联合稿、同集团镜像或共同上游稿件必须去重。
- Google News、Techmeme、GDELT 等聚合源只提供发现指针。
- Hacker News、Reddit、YouTube 评论、Bluesky、Mastodon 等社区/社交信息只提供发现或互动信号。
- 来源的 `nature` 和 `roles` 来自审核目录，事件处理程序不得根据标题或模型输出临时提升来源权级。

## 7. 跨来源事件匹配

事件匹配采用“强身份优先、确定性语义次之、模型仅处理边界候选”的顺序。合并必须保守；宁可暂时保留两个早期事件，也不能把两个不同事实错误合并。

### 7.1 第一层：强身份匹配

出现下列任一稳定身份时，可以生成强候选关系：

- 相同 Canonical URL；
- 相同最终原始发布 URL；
- 相同 GitHub 仓库、Release 或版本标识；
- 相同论文标识；
- 相同明确产品版本或官方公告标识。

聚合发现 URL 本身不能作为跨媒体独立身份；应优先使用解析后的原始发布者 URL。

### 7.2 第二层：确定性语义匹配

没有强身份时，只有同时满足以下条件才允许规则合并：

1. 两条内容的有效发布时间相差不超过 48 小时；
2. 至少共享一个非泛化实体，例如明确公司、产品、模型、人物、监管机构或项目名；
3. 动作类型一致，例如发布、融资、收购、监管、合作、故障或版本更新；
4. 去除媒体名、栏目名和常见标题后缀后，标题相似度达到经过固定样本验证的安全阈值；
5. 不存在相互冲突的版本号、交易对象、监管主体、产品名或动作对象。

`AI`、`model`、`chip`、`agent`、`startup` 等通用词不能单独构成共享实体。同一实体在同一时间窗口发生不同动作时必须保持为不同事件。

阈值不得只根据一次真实波次临时调整；修改阈值必须更新算法版本并补充误合并/漏合并回归样本。

### 7.3 第三层：MiniMax 边界辅助

MiniMax 只处理通过前两层生成、但规则无法明确判定的少量候选对。输入范围限制为：

- 两条标题；
- 发布时间；
- 来源名称与来源性质；
- 受限长度的摘要片段；
- 规则提取的实体和动作类型。

输出必须由 Pydantic 校验为：

- `same_event`；
- `different_event`；
- `uncertain`；
- 结构化理由和置信说明。

`uncertain`、超时、429、非法 JSON、结构修复失败或 MiniMax 完全不可用时，一律不合并。模型输出只能支持候选配对，不能修改来源角色、证据根和 `confirmed` 状态。

### 7.4 匹配可审计性

每次事件版本至少可追溯：

- 参与成员及其 RawItem ID；
- 命中的匹配层级和具体理由；
- 使用的聚类算法版本与阈值版本；
- 若调用模型，记录模型、目的、耗时、token 和校验结果；
- 去重后的证据根及去重理由；
- 尚未满足的确认条件和已知限制。

不得记录 API Key、Authorization、Cookie、数据库连接串或完整受版权保护正文。

## 8. 证据根与确认语义

### 8.1 证据根去重

下列情况只能计为一个证据根：

- 同一 Canonical URL 的不同发现入口；
- 同一发布者下的重复 URL、语言镜像或栏目镜像；
- 明确转载同一上游稿件；
- 联合发布但内容来自同一原始材料；
- 聚合条目与它所指向的原始报道；
- 多家媒体仅重复引用同一个未核验的匿名上游说法。

同一媒体的多篇跟进可以增加上下文，但不能自动增加独立专业媒体根数量。

### 8.2 状态判定

- `confirmed`：存在一个适用的官方一手根，或至少两个独立专业媒体根。
- `emerging`：存在发现/热度信号，但尚未满足确认条件。
- 现有其他事件状态继续沿用原语义，本阶段不扩展状态枚举。

一个早期事件在后续抓取中获得足够证据时，应创建同一 Event 的新版本并升级为 `confirmed`，不能创建一个重复事件。确认条件失效或来源归属发生变化时，也必须通过新版本重新评估。

### 8.3 展示用中文理由

确认理由固定为可解释模板，例如：

- “已由官方一手来源确认”；
- “已由 2 家独立专业媒体交叉确认”；
- “当前只有 1 家独立专业媒体，仍缺少 1 个独立证据根”；
- “当前仅有聚合/社区发现信号，尚无可独立确认的证据”。

模型可改善中文表达，但不能改变模板背后的计数和结论。

## 9. 指标与展示

### 9.1 统一运行指标

本阶段新增或补齐以下可计算指标：

- `evidence_capable_members`：冻结批次中具备证据角色且允许直接抓取的成员数；
- `direct_evidence_fetch_succeeded`：本轮成功抓取的直接证据来源数；
- `events_with_official_root`：含官方一手根的事件数；
- `events_with_one_professional_root`：恰有一个独立专业媒体根的事件数；
- `events_with_two_professional_roots`：至少有两个独立专业媒体根的事件数；
- `confirmed_event_count`：本轮最终确认事件数；
- `ambiguous_pairs_checked`：进入边界候选判断的配对数；
- `model_pair_fallback_count`：模型失败或不确定后按规则保守不合并的次数。

指标优先从现有冻结快照、抓取结果和事件版本推导；只有现有结构无法稳定复现时才允许增加最小字段或迁移。

### 9.2 网页

不新增导航和页面，只增强现有页面：

- 首页继续展示已确认、早期信号和趋势；
- 事件详情明确展示“官方确认”或“两个独立媒体确认”；
- 早期信号展示当前专业媒体根数量及仍缺少的条件；
- 运行页展示本轮 41 个目标、直接证据抓取成功数和确认事件数；
- 没有真实快照时继续显示明确的中文降级状态，不用演示数据冒充真实结果。

### 9.3 CLI 与报告

中文 CLI 报告与网页使用同一查询结果和状态映射，必须显示：

- 冻结目标和终态统计；
- 直接证据来源成功/失败/阻塞；
- 各类证据根事件数量；
- 已确认与早期事件数量；
- 边界候选和模型回退次数；
- 三轮结果对比、人工抽查结论和未通过原因。

## 10. 失败、并发与安全边界

- 单个专业媒体 RSS/API 失败不能阻塞整个批次。
- 单一专业媒体报道只能保留为 `emerging`，不能因其他聚合指针而升级。
- MiniMax 失败不得阻塞规则流程，也不得触发宽松合并。
- 401、403、429、超时、定义漂移、字段漂移和来源归属不清必须保留真实错误分类与中文解释。
- 运行继续复用现有 operation deadline、claim fencing、心跳、取消和安全恢复机制。
- RawItem、事件配对、事件版本和报告生成必须保持幂等，重复执行不得造成事件或证据根爆炸。
- 本阶段仅使用已经审核的公开 RSS/API/官方入口，不新增 HTML 回退。
- 日志、网页、报告和模型记录中不得出现密钥、认证头、Cookie、数据库凭据或完整正文。

## 11. 测试设计

### 11.1 Profile 与来源

- Profile 恰好包含 41 个唯一 Target。
- 6 个新增来源全部已完成初步接入审核、可直接访问、最近探测成功并包含 `evidence` 角色；深度研究状态不得被伪装为已完成。
- 受限、目录占位、间接发现或定义漂移来源不能因 Profile 配置进入内容抓取。
- 冻结批次不受随后 YAML 或数据库目录变化影响。

### 11.2 聚类与证据

- 官方自有发布可确认相关官方事实。
- 两个独立专业媒体根可以确认非官方事件。
- 聚合、社交、社区或单一专业媒体不能确认事件。
- 相同 Canonical URL、同发布者、转载和同上游稿件只计一个根。
- 相似实体但不同动作不能合并。
- 只有 `AI` 等泛化词相同不能合并。
- 强身份匹配、规则语义匹配和边界候选分别保存正确理由。
- 早期事件补充证据后升级同一事件版本，不产生重复 Event。

### 11.3 MiniMax 降级

- `uncertain`、非法 JSON、超时和 429 均不合并。
- 结构错误只允许一次修复重试。
- MiniMax 完全关闭时可完成抓取、聚类、证据判断和页面快照。
- 模型使用记录不包含敏感信息和完整正文。

### 11.4 Worker 与页面

- 单来源失败、取消、deadline、旧 claim 和恢复场景不阻塞批次终态。
- 重复运行不生成重复 RawItem、重复 Event 或重复证据根。
- 首页、详情、运行页和 CLI 对同一事件显示一致的证据数量与中文结论。

## 12. 三轮真实验收

使用冻结的 41 个目标连续执行三轮真实抓取。每轮必须记录：

- 冻结定义版本和访问方式；
- 成功、失败、阻塞和跳过数量；
- RawItem 新增、更新和重复数量；
- 事件新增、更新、合并和状态升级数量；
- 官方根、单专业媒体根、双专业媒体根和确认事件数量；
- 边界候选、模型回退和耗时；
- Worker 是否存在超时、卡住、旧 claim 覆盖或恢复失败。

验收条件：

1. 所有可抓取成员到达成功或明确失败终态，受限成员给出明确阻塞原因；
2. 三轮均无卡住批次、RawItem 重复爆炸或事件重复爆炸；
3. 至少一轮自然产生至少一个真实 `confirmed` 事件；
4. 每个确认事件都能追溯到官方根或两个独立专业媒体根；
5. 人工抽查最多 20 个事件或当轮全部事件（取较小者），重点检查误合并、转载去重和来源归属；
6. 首页、详情、运行页和中文报告对同一轮结果一致；
7. 关闭 MiniMax 后完成一轮完整快照；
8. 日志、网页和报告中不存在密钥、认证信息或完整正文泄漏。

若第 3 条未满足，但工程和安全测试均通过，必须将结果写为“数据验收未通过”，并分别报告：

- 证据型来源抓取成功率；
- 同事件跨来源候选命中数；
- 规则拒绝原因分布；
- 疑似漏合并样本；
- 是否需要进入 v2 的信号驱动定向证据搜索。

## 13. 实施边界与顺序

后续实施计划应按大步骤组织，避免再次横向重构：

1. 扩展 Profile 并锁定 41 个冻结成员；
2. 为新增来源补齐固定样本和直接抓取回归；
3. 增强确定性跨来源候选与聚类审计；
4. 接入受限的 MiniMax 边界配对降级；
5. 补齐证据指标及现有网页/CLI 展示；
6. 完成全量回归、三轮真实验收和最终审查。

每一步必须复用现有组件并在通过对应测试后进入下一步。除非现有持久化结构无法表达已经确认的审计事实，否则不得新增数据库对象。

## 14. 后续阶段（不属于本次实现）

若 v1 的工程能力稳定但真实数据仍长期缺少交叉证据，下一阶段再评估“信号驱动定向证据搜索”：以早期事件的实体、动作和时间窗口，定向查询已审核的专业媒体或搜索 Provider。该能力需要单独设计成本、限流、查询审计和误匹配边界，不能在本阶段顺手加入。

## 15. 模型建议

- 本设计规格、实施计划和最终安全/证据边界审查：`5.6 Sol + 高推理`。
- 按已批准计划进行常规实现与测试：`5.6 Terra + 中推理`。
- 涉及跨来源匹配阈值、证据独立性或真实验收结论的最终审查：切回 `5.6 Sol + 高推理`。
