# Event Intelligence v1 设计规格

日期：2026-07-12

状态：已确认设计，待实施计划

适用项目：News Codex

## 1. 背景与目标

News Codex 已具备来源注册、Provider/Target 覆盖、RawItem 抓取、持久化任务队列、Worker、失败恢复、中文来源感知台和运行诊断能力。现阶段系统可以回答“抓到了什么、来源是否健康”，但尚不能回答用户最关心的问题：“过去 24 小时最值得关注的 AI 事件是什么？”

Event Intelligence v1 在现有 RawItem 之上增加事件情报层。它不重做来源和抓取系统，而是把多来源原始信息转换为可追溯、可解释、中文优先的事件列表。

本阶段聚焦全球 AI/技术产品与研究热点，首页目标是在数据质量允许时展示过去 24 小时最值得关注的 10–20 个事件。事件数量不足时不以低质量内容凑数。

## 2. 产品原则

1. 规则主导，模型增强。确定性规则负责基本过滤、候选生成、证据门槛、评分和发布；MiniMax 负责语义归并、中文表达和规则遗漏的实体补充。
2. 证据优先。每个正式事件必须能追溯到 RawItem 和原始链接。
3. 热度与可信度分离。社交热度不能自动提高事实可信度。
4. 降级可用。MiniMax、单一来源或单个任务失败时，规则流程和网页仍能工作。
5. 运行可解释。评分、聚类、合并、拆分和中文生成都保留版本与依据。
6. 复用现有基础设施。继续使用当前数据库、Operation 队列、Worker、租约、心跳、恢复、取消、重试、审计和中文网页框架。

## 3. 范围

### 3.1 纳入范围

- AI 模型和产品发布。
- AI 开发工具、SDK、框架和重要开源项目变化。
- AI 研究论文、基准、数据集和研究机构动态。
- 与 AI 直接相关的重要公司动态。
- 社区和社交平台出现的早期线索与热度变化。
- 过去 24 小时重点事件首页、事件列表、事件详情和“正在升温”页面。
- 事件相关性、实体、聚类、证据角色、热度、可信度和重要度。
- MiniMax 中文标题、简短说明、关注理由和来源分歧解释。
- 事件处理任务的错误、日志、重试、取消、心跳、恢复和网页诊断。

### 3.2 明确不纳入

- 泛社会新闻或与 AI 关系弱的综合科技新闻。
- 个性化推荐。
- 日报、推送和重大提醒。
- 后台定时调度器。
- 长篇自动摘要或观点文章。
- 自动修改来源 YAML、来源合规结论或来源启用状态。
- 使用 Cookie、登录态爬虫或非官方高风险接口解锁受限平台。

## 4. 页面定位

新事件页面与现有来源感知台共存：

- `/`：过去 24 小时重点事件，作为日常阅读首页。
- `/events`：全部事件及分类、状态和时间筛选。
- `/events/<id>`：事件详情、时间线、证据、评分解释、分歧和原文。
- `/emerging`：只有社交或社区线索、尚未达到事实确认门槛的升温事件。
- `/sources`：保留现有来源感知台。
- `/operations`：保留并扩展任务、Worker、错误、重试和诊断。
- `/raw-items`：保留原始采集数据核验入口。

来源感知台继续回答“系统能看到哪里、怎么获取、是否健康”；事件首页回答“现在发生了什么、是否可信、为什么重要”。

## 5. 事件领域模型

### 5.1 EventCandidate

保存规则阶段形成、尚未发布的候选事件。候选包含处理版本、时间窗口、候选标题、主要实体、主题、成员 RawItem、相似度依据和候选状态。

### 5.2 Event

保存对网页可见的事件身份与当前版本。核心字段包括：

- 稳定事件 ID。
- 规范中文标题和原始语言标题。
- 类别与主题。
- `occurred_at`、`first_seen_at`、`last_updated_at`。
- 当前事件状态。
- 当前发布版本。
- 当前重要度、可信度、热度和 AI 相关性。

事件状态固定为：

- `emerging`：刚出现或证据不足。
- `confirmed`：有官方、研究或独立专业媒体证据。
- `developing`：事件持续产生重要更新。
- `disputed`：来源存在实质冲突。
- `stale`：离开当前关注窗口且无重要更新。
- `rejected`：广告、无关、误聚类或低质量传闻。

### 5.3 EventItem

保存 Event 与 RawItem 的多对多关系，包括：

- 证据角色。
- 是否为独立证据。
- 是否为聚合转载或同源引用。
- 加入与移出事件的版本。
- 聚类依据和置信度。
- 人工调整记录。

证据角色固定为 `official`、`professional_media`、`research`、`community`、`social`、`aggregator`。

### 5.4 EventEntity

保存公司、机构、人物、产品、模型、论文、数据集和项目等规范实体，以及原文提及、别名、实体类型和提取来源。

### 5.5 EventScore

每次评分保存不可变快照，包括规则版本、分项分数、总分、可读原因和计算时间。热度、可信度和重要度分别保存，不能只保存一个不透明总分。

### 5.6 EventVersion

记录标题、说明、状态、成员、评分、合并和拆分变化。网页读取完整的已发布版本，不读取正在更新的中间状态。

### 5.7 EventModelRun

记录调用目的、模型、提示版本、输入输出 Token、延迟、状态、错误分类和所生成的事件版本。不得保存 API Key。模型输出必须经过 Pydantic 校验。

## 6. 处理管线

管线使用现有 Operation 队列和 Worker，每个阶段是独立、可恢复、幂等的任务：

1. `filter_raw_items`：规则优先判断 AI 相关性，输出原因和规则版本。
2. `extract_entities`：规则提取公司、产品、模型、论文、机构和项目；MiniMax 可补充低置信度实体。
3. `cluster_events`：根据 Canonical URL、标题指纹、时间接近度、实体重合和主题重合生成候选簇。
4. `enrich_event`：MiniMax 辅助判断语义上是否属于同一事件，并生成中文标题、简短说明、关注理由和分歧解释。
5. `score_event`：确定性规则计算 AI 相关性、来源覆盖、权威性、时效、互动增长和新颖性。
6. `publish_event`：校验证据门槛，原子发布完整事件版本。

任务不得在一个长 Operation 中串行执行所有网络和模型调用。每个阶段先落库再触发下游，失败只影响对应对象和阶段。

## 7. 聚类规则

候选生成采用分层策略：

1. Canonical URL 或稳定外部 ID 相同：强匹配。
2. 规范标题指纹相同且时间接近：强匹配。
3. 核心实体、动作和时间窗口高度重合：候选匹配。
4. 同一官方公告的聚合转载或媒体引用：归入同一事件，但只计算一次根证据。
5. 同一公司在短时间内发布的多个细碎更新，只有在共享一个核心发布动作时才合并。
6. 论文、代码仓库和官方研究页面可归入同一研究事件，但必须保留各自证据角色。
7. 只有主题相似、没有共同动作或对象的内容不得合并。

MiniMax 只能对规则生成的候选对进行语义判断，不能在没有确定性候选依据时任意合并整个数据集。低置信度结果保持为独立候选，交由后续证据或人工操作处理。

## 8. 评分与发布门槛

重要度由以下分项构成：

- AI 相关性：25%。
- 来源覆盖：20%。
- 来源权威性：20%。
- 时效性：15%。
- 互动量与增长速度：10%。
- 新颖性：10%。

实现必须保存每个分项的原始输入、分数和可读解释。评分配置采用版本化常量或配置，不把权重硬编码在网页模板中。

可信度独立计算：

- 官方一手证据显著提高可信度。
- 两家互相独立的专业媒体可形成强确认。
- 研究预印本必须标记“未经同行评审”等已知限制。
- 社区和社交来源只贡献线索、热度和使用反馈。
- 聚合器、转载和共同引用同一根报道不算独立确认。
- 来源发生实质冲突时，事件进入 `disputed`，并展示冲突双方证据。

重点首页的正式事件必须满足：

- 发生时间或重要更新时间处于最近 24 小时。
- AI 相关性达到配置阈值。
- 至少有一条 `official`、`research` 或 `professional_media` 证据。
- 未被拒绝，且当前版本完整发布。

只有 `community` 或 `social` 的候选只进入 `/emerging`，不得以“已确认事实”进入重点列表。

## 9. 中文与 MiniMax 边界

第一版中文增强包含：

- 中文事件标题。
- 一句话说明“发生了什么”。
- 简短说明“为什么值得关注”。
- 来源冲突或证据限制说明。

模型使用：

- `MiniMax-M2.7-highspeed`：实体补充、候选对判断、批量翻译和短说明。
- `MiniMax-M3`：仅在复杂事件需要跨来源分歧解释时使用，不作为默认批处理模型。

所有原始内容在提示中标记为不可信数据，模型无工具、文件、数据库或环境变量访问权限。结构错误只允许一次修复重试；失败后回退到规则结果。

中文降级顺序：

1. 使用最近一个已发布且合法的中文版本。
2. 没有历史版本时使用规则标题或英文原标题。
3. 页面显示“中文增强暂不可用”，但不得无限加载或阻塞事件证据展示。

## 10. 并发、可靠性与日志

- 同一 RawItem、处理阶段和算法版本具有唯一幂等键。
- 同一事件使用事件级租约，防止两个 Worker 同时发布冲突版本。
- 不同 RawItem 和不同事件允许在配置的并发上限内并行处理。
- 网络和 MiniMax 调用期间不得持有数据库事务或行锁。
- 所有数据库更新使用短事务，并设置锁等待上限。
- MiniMax 具有独立连接、读取和总超时，以及并发上限、Retry-After 和指数退避。
- Worker 异常退出后，过期租约由其他 Worker 恢复。
- 单条坏数据、单个模型错误和单个事件失败不能中断批次。
- 网页只读取最后一个完整发布版本，不等待正在运行的后台任务。
- 错误记录必须关联 Operation、Attempt、Event、RawItem、处理阶段、错误分类和可读原因。
- 错误内容进入数据库和日志前必须执行凭据与 URL 查询参数脱敏。

现有操作页扩展事件任务筛选、失败阶段、重试关系和事件/RawItem 下钻。网页人工操作必须经过现有一次性操作令牌和审计服务。

## 11. 网页体验

### 11.1 首页

首页采用已经确认的 A 风格清晰信息层级，显示：

- 过去 24 小时窗口和最近更新时间。
- 重点事件、已确认、正在升温和存在分歧的数量。
- 综合排序、最新、可信度和热度增长筛选。
- 中文标题和一句话说明。
- 类别、状态、独立证据数和来源角色。
- 重要度、可信度和评分解释入口。
- 来源/Worker 异常的非阻塞状态提示。

### 11.2 事件详情

详情页显示：

- 事件时间线。
- 所有成员 RawItem 与原始链接。
- 官方、媒体、研究、社区、社交和聚合证据角色。
- 独立证据与同源转载关系。
- 重要度、可信度和热度分项解释。
- 分歧、限制和确认状态。
- 事件、规则、模型和聚类版本。
- 模型失败时使用的降级版本。

### 11.3 人工操作

网页提供受审计的重新聚类、重新生成中文、排除错误信息、合并事件和拆分事件操作。操作只创建队列任务或领域命令，网页进程不得直接执行网络或模型工作。

## 12. 测试与验收

### 12.1 自动测试

- 相关性规则的正例、反例和边界样本。
- URL、标题、实体、时间和主题聚类固定样本。
- 相同事件多来源归并和相似主题误合并防护。
- 同源转载、聚合引用和独立证据识别。
- 事件合并、拆分和版本历史。
- 研究预印本限制、来源分歧和纯社交线索状态。
- 评分分项、阈值和确定性重放。
- Operation 幂等、并发租约、重试、取消、超时、心跳和恢复。
- MiniMax 非法 JSON、超时、429、5xx 和完全不可用降级。
- 凭据、查询参数和错误信息脱敏。
- 网页操作安全、审计关系和 Worker-only 执行边界。

### 12.2 真实数据验收

- 使用现有 PostgreSQL 中的真实 RawItem 完成三轮连续事件处理。
- 每轮单个来源、单条坏数据或模型失败均不阻塞批次。
- 产品发布、研究论文、开发工具和公司动态四类均能形成可追溯事件。
- 在数据充分时，首页展示 10–20 个重点事件；不足时如实少于 10 个。
- 正式事件全部拥有原始链接和至少一条合格事实证据。
- 纯社交线索只出现在“正在升温”。
- MiniMax 完全关闭时仍能生成规则事件和可用网页。
- 中文首页、详情、异常、重试和人工操作完成浏览器验收。

### 12.3 完成条件

以下条件全部满足才算 Event Intelligence v1 完成：

1. 数据库迁移可升级、降级并通过现有数据兼容测试。
2. 完整测试套件、静态检查和迁移检查通过。
3. 三轮真实数据验收完成并生成可复核报告。
4. 首页与详情页能够解释事件、证据和评分。
5. 模型、来源和 Worker 故障不会造成网页永久等待或批次整体阻塞。
6. 操作、错误和事件版本可以从网页与日志中定位。
7. 不引入本规格范围外的推荐、推送、日报和调度功能。

## 13. 后续阶段

Event Intelligence v1 验收后，按以下顺序评估：

1. MiniMax 中文表达质量与事件分歧解释优化。
2. 后台调度和来源/事件管线健康告警。
3. 中文日报、重大事件提醒和事件时间线。
4. 用户主题、公司和研究方向偏好。
5. 在用户提供合规凭据后解锁 YouTube、Reddit 等受限来源。

后续阶段不得反向削弱事件证据、来源合规、模型降级和运行可追溯原则。
