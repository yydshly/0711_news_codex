# 日报可重复修订与回收站版本链设计

## 背景

日报详情页已经为已归档日报提供“创建修订版”入口。现有模型通过 `supersedes_report_id` 把版本串成链，并使用唯一约束保证同一父版本只有一个直接后继，从而让重复点击保持幂等。

当前缺陷是：修订草稿移入回收站后仍占用父版本的唯一后继位置。用户再次创建修订版时，仓储层会复用回收站记录，网页随后重定向到回收站，无法形成新的可用草稿。单纯恢复旧草稿会丢失“用户已经放弃该草稿”的语义，也不能解决以后重复发生的问题。

此外，最近加入的“从固定事件快照重建全览”要求来源操作必须存在完整事件快照。旧数据或测试夹具缺少快照时，网页修订入口会返回 409。修订流程需要在不重新抓取、不改写归档版的前提下，提供明确且安全的兼容路径。

## 目标

- 保留现有修订入口，作为长期维护已归档日报的正式能力。
- 归档日报和已生成音频保持不可变；所有修改通过新草稿完成。
- 同一操作被重复提交或并发提交时只产生一个有效草稿。
- 修订草稿进入回收站后，可以创建新的有效修订版，不复用回收站记录。
- 用户可以从版本链中的任意已归档页面发起修订，系统自动定位最新有效版本。
- 固定事件快照存在时重建完整全览；快照缺失时安全复制归档版已有固定条目，并记录中文降级诊断。
- 恢复、永久清理和版本链重挂接在出现废弃分支后仍保持一致。

## 不在范围内

- 不重新抓取来源，不重新运行 41 个高价值目标。
- 不改变来源合法性、证据确认或事件可信度结论。
- 不允许 MiniMax 决定是否收录或是否合法。
- 不覆盖、删除或原地修改已归档日报。
- 不为一次修复直接修改生产数据库中的 #17 或 #18。
- 不重做日报页面或引入第二套日报模型。

## 方案比较

### 方案 A：自动恢复并覆盖回收站草稿

改动最少，但会把用户已经删除的草稿静默恢复，并可能覆盖其历史内容；多次失败时缺少清晰审计记录，不采用。

### 方案 B：保持全局单后继约束，让新草稿继续挂在回收站草稿之后

不需要迁移，但有效版本链会包含被删除的中间节点。旧页面需要穿越回收站节点才能找到最新版本，恢复与永久清理语义复杂，不采用。

### 方案 C：每个父版本只限制一个“未删除的有效后继”（采用）

把后继唯一约束改为仅作用于 `deleted_at IS NULL` 的部分唯一索引。回收站草稿保留审计记录但不再占用有效后继位置；同一个父版本历史上可以存在废弃草稿和一个有效草稿，但任意时刻最多只有一个有效后继。配合最新有效版本解析、事务锁与冲突后复查，可兼顾幂等、并发安全、历史保留和重新生成。

## 数据模型与迁移

1. 保留 `(report_date, window_hours, revision)` 唯一约束，版本号只递增，不复用回收站版本号。
2. 将 `uq_daily_report_supersedes` 改为部分唯一索引：`supersedes_report_id IS NOT NULL AND deleted_at IS NULL`。
3. 将根日报身份索引 `uq_daily_report_identity` 同样限制为 `supersedes_report_id IS NULL AND deleted_at IS NULL`，避免根草稿进入回收站后永久阻塞同一固定操作的显式重建。
4. PostgreSQL 和 SQLite 迁移保持等价；降级迁移先检测并拒绝已有冲突的有效记录，不能静默丢数据。
5. 回收站记录继续保留 30 天，永久清理后版本号也不回收。

## 版本解析与生成规则

服务层新增“解析有效修订目标”的单一职责逻辑：

1. 从用户打开的日报开始，沿未删除后继查找最新有效节点。
2. 如果最新有效节点是草稿，直接返回该草稿；重复点击不创建新记录。
3. 如果最新有效节点已归档，为它创建一个新草稿。
4. 如果原父版本只有回收站后继，则从该父版本创建新的有效草稿，版本号使用当前日期和窗口下的最大版本号加一。
5. 如果数据库唯一约束报告竞争冲突，事务回滚后重新查询有效后继并返回获胜记录；不得返回回收站记录。
6. 每次创建都复用现有 PostgreSQL advisory lock；SQLite 依靠唯一索引与有限冲突重试。

这套规则适用于从任意历史归档页面点击入口。网页无需要求用户先找到最新版本。

## 固定快照与兼容降级

- 存在完整事件操作快照时，修订版根据原日报的 `source_operation_id` 和 `generated_at` 重建全览，以纳入当时固定快照中的全部确认、热点和信号条目。
- 快照缺失时，不重新抓取，也不读取当前不断变化的事件状态；改为复制原归档日报已有的决策条目、全览条目、顺序和审核关系。
- 降级创建成功后，在 `generation_summary` 记录稳定代码 `revision_overview_source=archived_report_snapshot` 和中文诊断，页面明确说明“历史操作快照缺失，本修订版沿用归档版固定条目”。
- 快照存在但结构损坏时仍拒绝创建，并返回可定位的中文错误，不能把真正的数据损坏伪装成兼容降级。

## 回收站、恢复与永久清理

- 移入回收站只改变保留状态，不修改正文、来源操作、版本号或审核历史。
- 如果某个回收站草稿的父版本已经有新的有效后继，直接恢复会违反有效单后继规则。恢复操作应返回中文冲突诊断，不自动覆盖新版本。
- 永久清理废弃草稿时，只处理指向它的真实后继；同一父版本的其他有效或废弃兄弟版本不应被改写。
- 清理中间版本时继续使用现有过渡表和触发器，但查询必须从“至多一个子节点”的假设改为按具体子节点处理，并增加多子节点回归测试。
- 运行中的自动日报、音频或清理任务仍阻止删除，现有取消与结构化诊断保持不变。

## 页面行为

- “创建修订版”继续只在已归档日报详情页显示；草稿显示“归档定稿”。
- 从旧归档版点击时，如果已有更新的有效草稿，重定向到该草稿，并显示“已打开当前修订草稿”。
- 回收站旧草稿不会成为修订入口的重定向目标。
- 恢复发生版本冲突时停留在回收站并显示中文原因和下一步动作。
- 不新增页面，只扩展现有详情页、回收站页和操作提示。

## 多次生成的确定行为

| 场景 | 结果 |
| --- | --- |
| 连续点击两次创建修订版 | 返回同一个有效草稿 |
| 两个请求并发创建 | 只有一个有效草稿，失败方复用获胜记录 |
| 有效草稿已在回收站 | 创建下一版本号的新草稿 |
| 从旧归档版点击，链上已有有效草稿 | 打开最新有效草稿 |
| 最新修订版已经归档 | 从最新归档版创建下一版 |
| 同一固定操作重复生成根日报 | 复用未删除根草稿或归档版 |
| 同一固定操作的根日报已在回收站 | 允许显式重建新根日报 |
| 创建事务中途失败 | 整体回滚，不留下半份日报或半套条目 |

## 测试与验收

严格按测试驱动实现，先观察失败再写生产代码：

1. 仓储测试：回收站后继不被复用，新有效后继使用更大版本号。
2. 仓储测试：有效后继仍被重复请求复用。
3. 服务测试：从旧版本解析到最新有效草稿；归档后继续生成下一版。
4. 服务测试：完整快照重建全览；缺失快照复制归档固定条目并记录降级原因；损坏快照继续拒绝。
5. 恢复测试：存在有效兄弟版本时返回稳定冲突，不修改任何记录。
6. 清理测试：父版本存在一个有效后继和一个或多个回收站后继时，永久清理不破坏有效链。
7. 迁移测试：SQLite 和 PostgreSQL 索引谓词、升级数据保持及降级保护正确。
8. 并发测试：PostgreSQL 同时提交只形成一个未删除有效后继。
9. 网页测试：入口长期存在、重复点击正确重定向、回收站版本不被打开、中文诊断可理解。
10. 完成后运行完整 `pytest`、`ruff check .`，并在真实本地网页上验收创建、重复点击、回收站重建、恢复冲突和继续下一版。

当前真实数据验收以 #17/#18 为样例：#17 保持归档，#18 保持回收站；修复后从 #17 创建新版本，应使用 #17 的固定事件操作快照重建全览，重复点击只能打开同一个新草稿。未经用户确认不合并、不推送。
