# News Codex 广覆盖来源试用设计

## 背景与目标

News Codex 的首要价值是让用户尽快获得更广的全球 AI/技术信息，而不是先把每一个来源做成完整合规档案。现有来源地图已经登记 166 个具体 Target，但其中只有少量完成了严格研究资料；这不应阻止可公开获取的信息进入试用。

本阶段目标是把来源管理拆成“首次探索与试用”和“后续深度审计”两条节奏：先批量探测、先获得真实信息，再依据运行价值决定哪些来源值得长期投入。

## 范围

本阶段覆盖现有 YAML 来源目录中的全部 166 个 Target，并按现有 `coverage_mode` 与 `availability` 分流：

- `direct`：公开 RSS、Atom、REST 或公开 API 入口。
- `indirect`：聚合、搜索或发现入口，内容必须标记为需要回到原始来源确认。
- `catalog_only`：受限平台、邮件型通讯、未确认入口或人工目录，仅展示覆盖缺口。

不实现邮件收取、登录态/ Cookie 爬取、代理绕过、验证码绕过、付费内容读取、音频下载或字幕抓取。不把 X、Facebook、Instagram、TikTok、LinkedIn 等受限平台因为不能免费读取而从来源地图中移除。

## 核心概念

### 来源生命周期

每个 Target 同时拥有“目录状态”和“运行资格”，两者不能混淆：

1. **已登记**：来源存在于地图，可能尚未探测。
2. **已探索**：至少完成一次真实能力或内容探测，并记录时间、结果、字段完整度与失败分类。
3. **试用可抓取**：满足公开直连、首次探测成功、基础字段可用、未触发硬阻断；允许进入受控 Worker 抓取。
4. **仅发现**：聚合或社交信号可用于发现候选事件，但不得单独确认事实。
5. **受限目录**：需要凭据、审批、付费或仍待确认；可视化展示解锁条件，不进入抓取。
6. **长期来源**：经后续深度审计与连续稳定性验证后，再决定是否长期保留。

“研究资料已齐”是后期审计标签，不再作为首次试用的前置条件；“试用可抓取”也不等于事实已确认或长期稳定。

### 三条处理路径

| 路径 | 进入条件 | 运行方式 | 输出标记 |
| --- | --- | --- | --- |
| 直连试用 | `coverage_mode: direct`、`availability: ready`、首次探测成功、基础字段可用 | Worker 受控抓取，遵守现有频率与限流 | `trial_fetch` |
| 间接发现 | `coverage_mode: indirect` 且探测成功 | 只生成发现候选；事件必须回到官方或独立专业媒体确认 | `discovery_only` |
| 受限目录 | `catalog_only` 或需要凭据、审批、付费 | 不抓取；展示解锁步骤、风险和缺口 | `catalog_only` |

## 阶段 A：全量初探与试用池

1. 对所有 166 个 Target 运行其允许的协议探测。单个失败不得中断批次。
2. 对可直接访问的来源收集：HTTP 状态、内容类型、样本数、字段完整度、最新内容时间、分页/限流信息、失败分类与探测时间。
3. 首次成功的直连来源进入“试用候选池”；首次失败的来源保留目录和可读失败原因。
4. 只对完成同步且 `ingestion.enabled` 显式允许的来源投递 Worker；不能以探测成功自动绕过现有来源 YAML 的人工边界。
5. 生成中文覆盖报告：总量、已探索、试用候选、间接发现、凭据/审批/付费阻塞、失败及未确认入口。

首次探测的最低可用门槛为：探测器成功、至少有标题和规范 URL、样本数大于零、字段完整度达到 60%。发布时间缺失的社区/聚合来源可以保留为发现信号，但不得被标为试用事实来源。

## 阶段 B：依据真实使用结果筛选

试用池运行后，以真实结果而非预设声誉筛选：

- 新内容频率与最近更新时间。
- 字段稳定性、重复率、限流与失败频率。
- 是否带来其他来源没有的 AI/技术热点。
- 来源角色是否平衡：官方、专业媒体、研究、社区和聚合发现均需有代表。

输出三类结论：扩大试用、保留为发现信号、暂停。该阶段不生成新闻摘要、不做推荐算法，也不使用 MiniMax 决定来源合规性。

## 阶段 C：按价值深度审计

只有高价值或高风险来源进入深度审计。审计补齐官方身份、内容权利、备用方法、认证/付费条件、字段漂移、连续探测与风险结论。通过后才可能升级为长期来源。

需要 GitHub、Reddit、YouTube 等凭据的平台，先在网页和报告中显示中文解锁说明；用户未配置凭据时保持阻塞，不降级为网页登录抓取。

## 页面与可观察性

网页必须让用户区分：

- “已登记多少”与“正在抓取多少”。
- “试用可抓取”与“长期来源”。
- “可直接获取”与“仅发现/待回源确认”。
- “来源不可用”与“需要你提供凭据、审批或预算”。

每次批量初探都保留运行记录、每个来源的中文原因和非敏感错误分类。页面、报告和日志不得显示 API Key、Cookie、代理地址、`.netrc` 内容或数据库连接串。

## 验收标准

- 166 个 Target 均有可见的首次探索状态或明确的受限原因。
- 免费直连来源以真实探测结果形成试用候选清单，不因未完成深度审计而被排除。
- 间接来源全部标识为发现信号，不能单独成为事实证据。
- 受限平台及邮件型来源仍在覆盖视图中，且没有任何登录态或邮件接入。
- Worker 的试用抓取失败可追踪，单个来源失败不阻塞其他来源。
- 深度审计与三轮稳定性仍被保留为后续长期来源的升级条件，不阻塞阶段 A。

## 非目标

- 不在本阶段实现中文新闻摘要、推荐、推送、定时常驻调度或新前端重构。
- 不变更 MiniMax 的职责：它可以后续辅助分类和异常解释，但不决定来源是否合规、是否启用。
